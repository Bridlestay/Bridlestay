## Session 1: Route creation UX improvements (6 fixes)

### Summary
Six improvements to route creation and map interaction: cluster navigation scoped to clicked cluster, route pins hidden during creation, cursor improvements, zoom-aware waypoint snap radius, waypoint numbering by proximity to start, and waypoint editing after placement.

### Files Modified
- `app/routes/page.tsx` ‚Äî Disabled visible routes expansion in cluster browse, added `distanceAlongCoords` helper, sorted POIs by route distance before saving with `order_index`, added POI edit state + handlers
- `components/routes/routes-map-mapbox.tsx` ‚Äî Hide/restore route pins and cluster layers on creation mode toggle, added per-tool cursors (crosshair/not-allowed/copy), added `distanceAlongRoute` helper, sorted POI marker numbering by route distance, zoom-aware snap tolerance (50m‚Äì200m), added `onCreationPOIEdit` callback for clicking POI markers
- `components/routes/quick-add-waypoint-dialog.tsx` ‚Äî Added edit mode: `editingWaypoint` and `onUpdate` props, pre-fills form, shows "Edit Waypoint" / "Save Changes" labels

### What Changed and Why

**1. Cluster navigation scope**: Previously, after clicking a cluster, the navigation expanded to include all visible unclustered pins as you zoomed/panned. This was confusing ‚Äî you'd see "3 of 12" when you only clicked a cluster of 3. Now cluster browse is locked to the originally clicked cluster's routes.

**2. Hide pins during creation**: Existing route pins and cluster circles stayed visible when entering creation mode, cluttering the map. Now all pin markers are removed and cluster layers hidden when `isCreating` becomes true, and restored when leaving creation mode.

**3. Cursor per tool mode**: Plot mode now shows `crosshair`, erase shows `not-allowed`, insert shows `copy`. Previously only plot had a crosshair but it could be overridden by hover events on the now-hidden pin layers.

**4. Zoom-aware waypoint snap**: The snap tolerance for placing waypoints on the route line was fixed at 50m. At low zoom levels, this made it very hard to click on the line. Now: 50m at zoom ‚â•14, 100m at zoom ‚â•12, 200m below that.

**5. Waypoint numbering by proximity**: POI waypoints were numbered W1, W2, W3 etc. by creation order. Now they're numbered by distance along the route from start. If you add a waypoint near the beginning after already adding one near the end, it correctly becomes W1. This applies both to map marker labels (during creation) and to `order_index` when saving to the database.

**6. Edit waypoints after placing**: Clicking a POI marker during creation (in any mode except erase) now opens the quick-add dialog pre-filled with the waypoint's name, tag, icon, and description. Changes are saved back to the creation state.

## Session 2: Fix cursor enforcement + cluster navigation includes all visible routes

### Summary
Two fixes: (1) Tool cursors (crosshair, not-allowed, copy) weren't actually showing because Mapbox's internal event handlers reset the cursor on every mouse interaction. (2) Cluster navigation now expands to include ALL visible routes (both unclustered pins and routes inside visible clusters) as you zoom/pan.

### Files Modified
- `components/routes/routes-map-mapbox.tsx` ‚Äî Cursor enforcement via persistent `mousemove` handler; `notifyVisibleRoutes` function expands visible clusters via `getClusterLeaves` to gather all route IDs in viewport
- `app/routes/page.tsx` ‚Äî Restored `handleVisibleRoutesChange` to expand cluster browse with all visible routes (capped at 50)

### What Changed and Why

**Cursor not showing**: The useEffect set the cursor once, but Mapbox's internal `mouseenter`/`mouseleave`/`dragPan` handlers immediately reset it to empty or "pointer" on any mouse movement. Fixed by adding a persistent `mousemove` event listener that continuously re-applies the cursor during creation mode. The listener is cleaned up when the tool mode changes or creation ends.

**Cluster navigation scope**: The previous fix went too far ‚Äî it locked navigation to only the clicked cluster's routes. The user wants to navigate ALL visible routes including those inside nearby clusters. Restored the `handleVisibleRoutesChange` expansion, and updated the map's `notifyVisibleRoutes` function to expand visible cluster features via `getClusterLeaves`, so the parent receives IDs from both unclustered pins and clustered routes.

## Session 3: Preserve original spine points when editing routes

### Summary
When editing a route, the map was showing dozens of intermediate coordinates from the snapped road geometry instead of the original few spine points the user placed. Fixed by saving the user's original spine waypoints as `spine_points` in the geometry JSONB, and using those when loading for edit.

### Files Modified
- `app/routes/page.tsx` ‚Äî Save `spine_points` in geometry payload; load from `spine_points` when entering edit mode
- `app/routes/edit/[id]/page.tsx` ‚Äî Same: prefer `spine_points` over `coordinates` when extracting waypoints
- `app/api/routes/route.ts` ‚Äî Added `spine_points` to zod validation schema

### What Changed and Why
**Root cause**: When saving a route with snap enabled, `getRouteGeometry()` returns the full road-following polyline with a coordinate for every bend in the road (e.g. 60+ points for a simple route). When loading for edit, every coordinate became a draggable spine point, making the route nearly uneditable.

**Fix**: The save payload now includes `spine_points: waypoints.map(wp => [wp.lng, wp.lat])` alongside the full geometry coordinates. When loading for edit, both `startEditing()` and the standalone edit page check for `geometry.spine_points` first, falling back to `geometry.coordinates` for routes saved before this fix.

3. **Old routes**: Edit a route saved before this fix ‚Üí still works (falls back to full coordinates)
4. **Route display unchanged**: The actual route line on the map still uses the full snapped geometry

## Session 4: Fix circular route reversion + snap drag fallback + edit spine_points

### Summary
Three fixes: (1) Deleting a waypoint in a circular route now reverts to linear if the remaining last point is >50m from the start. (2) Dragging a spine point with snap enabled now falls back to self-snapping to the nearest road when the neighbor-routing snap fails. (3) Edit page now saves `spine_points` so subsequent edits preserve original spine data.

### Files Modified
- `app/routes/page.tsx` ‚Äî Added `haversineSimple` helper; `removeWaypoint` now checks circular reversion (distance check + min 3 waypoints)
- `app/routes/edit/[id]/page.tsx` ‚Äî Added `haversineSimple` helper; `removeWaypoint` now checks circular reversion; save handler includes `spine_points` in geometry payload
- `components/routes/routes-map-mapbox.tsx` ‚Äî Drag handler fallback: when neighbor-routing snap fails (>25m), tries a self-snap via tiny-offset directions call (50m threshold)

### What Changed and Why

**1. Circular route not breaking on point removal**: `removeWaypoint` in both the creation and edit pages simply removed the waypoint without checking whether the route should revert from circular to linear. Now after removal, if the route is circular, it checks: (a) fewer than 3 remaining points ‚Üí linear, (b) last point >50m from first ‚Üí linear. On the creation page, also re-enables plotting mode.

**2. Snap drag floating disconnected**: The drag handler snaps by routing from a neighbor to the dragged position. If the snapped endpoint was >25m from the drag position (user dragged off-road), it kept the raw unsnapped position ‚Äî leaving the point floating disconnected. Now it tries a fallback: a tiny-offset directions call at the dragged position itself, snapping to the nearest road within 50m. This finds the closest road even when the neighbor-routing approach overshoots.

**3. Edit page missing spine_points**: The edit page's save handler sent `geometry.coordinates` but not `geometry.spine_points`. After editing and saving, the spine data was lost ‚Äî the next edit would show full snapped coordinates instead of the original spine points. Now saves `spine_points` alongside coordinates.

### Verification Steps
1. `npm run build` passes

3. **Snap drag**: Enable snap ‚Üí drag a spine point off the road ‚Üí it snaps to the nearest road instead of floating disconnected
4. **Edit spine preservation**: Edit a route ‚Üí save ‚Üí edit again ‚Üí only original spine points shown (not full road coordinates)

## Session 7: Route panels ‚Äî centered modal overlay (replacing side panels)

### Summary
Replaced the side panel layout for Find Routes and Saved Routes with centered modal overlays matching the main route detail card's sizing and style. Cards now display in a responsive 2-column grid inside the overlay.

### Files Modified
- `components/routes/saved-routes-panel.tsx` ‚Äî Complete rewrite: side panel ‚Üí centered modal overlay (`max-w-2xl max-h-[85vh]`, `bg-black/40 backdrop-blur-sm` backdrop, `rounded-2xl shadow-2xl`), sticky header with title/search/close, scrollable content with sort/tabs/filters/grid, 2-column responsive card grid, removed RoutesPanelHeader/MobileTopHeader/MobilePanelToggle dependencies
- `components/routes/find-routes-panel.tsx` ‚Äî Same centered modal rewrite plus: collapsible filters toggle (pill button with active count badge), filters in a subtle `bg-gray-50/50` section, removed RoutesPanelHeader/MobileTopHeader/MobilePanelToggle dependencies

### What Changed and Why

**User feedback**: Route panels should not be side panels ‚Äî they should be centered cards matching the main route detail card's exact sizing, shape, and style.

**Layout change**:
- **Before**: `absolute top-0 left-0 bottom-0 w-96` side panel on desktop, full-screen sliding panel on mobile
- **After**: `fixed inset-0 z-50 flex items-center justify-center p-4` centered overlay with `max-w-2xl max-h-[85vh] md:max-h-[80vh]` ‚Äî exactly matching the route detail drawer
- Backdrop: `bg-black/40 backdrop-blur-sm` (same as detail card)
- Animation: `animate-in zoom-in-95 slide-in-from-bottom-4 fade-in duration-300` (same as detail card)

**Header simplification**: Replaced the heavy RoutesPanelHeader (hamburger menu, profile avatar, separate search) with a minimal sticky header (title, close button, inline search). The hamburger menu and profile are already accessible via the main route nav tabs.

**Filters (Find Routes)**: Instead of showing all filters at all times (consuming half the viewport), filters are now collapsible behind a "Filters" pill button. The pill shows the count of active filters with a green badge. Expands to show distance slider, difficulty pills, rating pills, route type checkboxes, and "Routes Near Me" button.

**Card grid**: Route cards now display in a responsive `grid-cols-1 md:grid-cols-2` layout with `gap-4`, giving a proper grid browsing experience instead of a narrow single-column list. Cards have a subtle lift effect on hover (`hover:-translate-y-0.5`).

**Removed dependencies**: No longer imports RoutesPanelHeader, MobileTopHeader, or MobilePanelToggle ‚Äî the centered modal works identically on both desktop and mobile.

### Risk Notes
- The `mobilePanelOpen` and `onMobilePanelToggle` props are still accepted but no longer used (kept for backward compatibility with parent). These can be cleaned up later.
- RoutesPanelHeader, MobileTopHeader, and MobilePanelToggle components are no longer imported but still exist in the codebase. They may be used elsewhere (e.g., waypoint panel) ‚Äî do not delete them.
