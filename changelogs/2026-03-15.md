# Changelog — 2026-03-15

## Session 1 (Morning)

### Summary
Three changes: faster mini card counts (count-only API mode), 1.5s pin refresh, and the "I've ridden this route" ride tally system with 24h cooldown — moved to the bottom bar between Save and Navigate.

### Changes

#### Performance: Mini card count loading speed
- **Files modified**: `app/api/routes/[id]/nearby-properties/route.ts`, `app/api/routes/[id]/variants/route.ts`, `components/routes/route-quick-card.tsx`
- **What changed**: Added `?count_only=true` query parameter to both the nearby-properties and variants GET endpoints. When set, the API returns just `{ count: N }` using a lightweight `head: true` count query — skipping all data fetching, joins, photo lookups, and Haversine distance calculations. The mini card now uses these count-only endpoints instead of fetching full data just to call `.length`.
- **Why**: Mini card property/variant counts were slow because each card made 2 API calls that fetched full property objects with photos and full variant route objects with geometry — just to count them.

#### Performance: Pin refresh interval reduced to 1.5 seconds
- **File modified**: `app/routes/page.tsx`
- **What changed**: Map pin auto-refresh interval changed from 5 seconds to 1.5 seconds.
- **Why**: User requested faster pin refresh for a more responsive explore experience.

#### Feature: Ride tally system with 24h cooldown
- **Files created**: `supabase/migrations/079_ride_tally.sql`, `app/api/routes/[id]/ride/route.ts`
- **Files modified**: `components/routes/route-detail-drawer.tsx`

**Database migration** (`079_ride_tally.sql`):
- Added `ride_count INT DEFAULT 1` and `last_ridden_at TIMESTAMPTZ` to `route_completions`
- Added `total_rides INT DEFAULT 0` to `routes` table
- Added `total_rides INT DEFAULT 0` to `users` table
- Added UPDATE RLS policy on `route_completions` for ride count increments
- Created triggers: `update_route_total_rides` and `update_user_total_rides` to keep counts in sync
- Backfill for existing data

**API endpoint** (`POST /api/routes/[id]/ride`):
- GET: Returns ride status (hasRidden, rideCount, canRideAgain, cooldownEndsAt, hasReview)
- POST: Logs a ride tally — creates completion on first ride, increments ride_count on subsequent rides
- Server-side 24h cooldown enforcement (returns 429 if cooldown active)

**UI — Bottom bar redesign**:
- Moved "I've ridden this route" button from scrollable content to the bottom action bar
- 3-button layout: `[Save]` `[Ridden]` `[Navigate]`
- Button states:
  - Never ridden → green filled "Ridden" with checkmark → opens review form (non-owners) or logs ride directly (owners)
  - Cooldown active → grey disabled with clock icon showing countdown (e.g. "14h 32m")
  - Ready to tally → green outline "Ridden 5x" with checkmark → increments count
- Live countdown timer updates every minute during cooldown
- Ride status refreshed after review flow completes
- Owners can log rides too (skip review form on first ride)

## Session 3 (Afternoon)

### Summary
Route navigation feature — locate-me GPS button on map controls, blue pulsing user dot, follow mode with heading rotation, enabled Navigate button in detail drawer, wake lock during navigation, offline connectivity banner, and auto-complete ride tally when reaching the route endpoint.

### Changes

#### Feature: Locate-me GPS button on map controls
- **File modified**: `components/routes/map-layer-controls.tsx`
- **What changed**: Added a crosshair button above the layers icon. Tapping starts GPS watching and shows a blue pulsing dot on the map at the user's location. Tapping again re-centres the map. Button glows green when GPS is active.

#### Feature: User position dot on map
- **File modified**: `components/routes/routes-map-mapbox.tsx`
- **What changed**: Added a useEffect that renders a blue dot Mapbox marker when `userPosition` is set. Dot has a pulsing ring animation. During navigation (`followUser=true`), the map auto-centres on the user and rotates to their heading (bearing) at zoom 16+.

#### Feature: Navigate button enabled
- **File modified**: `components/routes/route-detail-drawer.tsx`
- **What changed**: The disabled Navigate button in the bottom bar is now functional. Added `onStartNavigation` prop — clicking Navigate closes the drawer and starts navigation mode. Button is green and active when the prop is provided.

#### Feature: Navigation flow wiring
- **File modified**: `app/routes/page.tsx`
- **What changed**:
  - `handleLocateMe` now uses `watchPosition` for continuous GPS tracking (not just `getCurrentPosition`)
  - `isLocating` state tracks whether GPS is active
  - `handleStartNavigation` fetches route waypoints and active hazards, sets up navigating state, closes drawer, starts GPS
  - `handleNavigationComplete` receives real stats from the navigator (distance, duration, speed) instead of hardcoded values
  - All three MapLayerControls instances receive `onLocateMe` and `isLocating` props

#### Feature: RouteNavigator improvements
- **File modified**: `components/routes/route-navigator.tsx`
- **What changed**:
  - **Wake lock**: Screen stays on during navigation (Screen Wake Lock API)
  - **Offline detection**: Amber banner "Limited Connectivity" when internet drops — GPS still works
  - **Auto-complete ride tally**: When rider reaches within 30m of endpoint, auto-calls `POST /api/routes/[id]/ride` to log the ride
  - **Real stats**: `onComplete` now passes actual distance, duration, and avg speed to the parent
  - **Completion guard**: Prevents duplicate completion events

### Files Modified

### Risk Notes
- GPS battery drain: wake lock + continuous GPS tracking. Consider showing battery warning in future
- Heading data (`coords.heading`) may be null on some devices — falls back to 0 (north)
- Web browsers pause GPS when tab is backgrounded. Wake lock helps but isn't 100% reliable
- Auto-tally fires when within 30m of endpoint — could trigger on GPS drift if near the end. The `completed` state guard prevents duplicate tallies
