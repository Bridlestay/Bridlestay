# Changelog — 2026-03-14

## Session 1 (Afternoon)

### Summary
Fixed elevation data not loading for longer routes, implemented property pin markers on the Mapbox map, and investigated bridleway map styling.

### Changes

#### Fix: Elevation data failing for longer routes
- **File modified**: `app/api/routes/[id]/elevation/route.ts`
- **What changed**: The Open-Meteo elevation API was called with all 200 sampled coordinates in a single GET URL, which could exceed URL length limits (4-5KB+). Switched to batched requests of 80 coordinates each, ensuring reliable elevation data for routes of any length.
- **Why**: Routes over ~5km were showing "--- m" for ascent/descent because the single large GET request was silently failing. Shorter routes worked because they had fewer sampled coordinates.
- **Risk**: Low — batched requests are functionally identical, just split across multiple API calls.

#### Feature: Property pins now render on Mapbox map
- **File modified**: `components/routes/routes-map-mapbox.tsx`
- **What changed**: The `propertyPins` prop was accepted by the Mapbox map component but never rendered — the toggle in map settings toggled state but nothing appeared on the map. Added a `useEffect` that renders green circle markers (with house icon) for each property, with click popups showing name, location, price, rating, and horse capacity.
- **Why**: The "Properties" toggle in map layer controls was non-functional. Data was already being fetched from `/api/properties/nearby` and passed to the map component.
- **Risk**: Low — additive feature, follows same marker pattern as existing POI markers.

#### Investigation: Bridleway map styling
- **No code changes** — the dotted yellow/orange lines visible on the map are from the Mapbox Outdoors base map style (`outdoors-v12`), not Padoq's custom bridleway overlay. Padoq's bridleways are shown in brown (#8B4513) via custom data layers loaded from OpenStreetMap.

### Files Modified
- `app/api/routes/[id]/elevation/route.ts` — Batched elevation API calls
- `components/routes/routes-map-mapbox.tsx` — Added property pin marker rendering

## Session 2 (Afternoon)

### Summary
Fixed cluster navigation arrows missing on desktop, and fixed edit route creating copies instead of updating the original.

### Changes

#### Fix: Restore cluster navigation arrows on desktop
- **File modified**: `components/routes/route-quick-card.tsx`
- **What changed**: The carousel redesign removed chevron arrows, leaving only dot indicators on desktop. Desktop now has dedicated prev/next arrow buttons flanking the card, plus a "1 of N" page indicator. Mobile keeps the swipeable carousel with dot indicators.
- **Why**: Desktop users had no discoverable way to navigate between clustered routes after the mobile carousel redesign.
- **Risk**: Low — desktop and mobile rendering are now cleanly separated with `md:hidden`/`hidden md:flex`.

#### Fix: Edit route was creating copies instead of updating
- **Files modified**: `app/routes/page.tsx`, `components/routes/save-route-modal.tsx`
- **Root cause**: When `startEditing` set `activeTab` to "create", a `useEffect` watching `activeTab` called `startCreating()`, which reset `isEditing=false`, `editingRouteId=null`, and `editingRouteData=null`. This wiped the editing state, so the save modal opened blank and used POST (create) instead of PATCH (update).
- **Fix in page.tsx**: Added `if (!isEditing)` guard in the tab-change effect so `startCreating()` is skipped when already editing.
- **Fix in save-route-modal.tsx**: Added `useEffect` to sync form state when `existingData` changes, as a safety net against timing issues.
- **Risk**: Low — the guard is a simple boolean check. The useEffect sync is a defensive measure.

### Files Modified

- `app/routes/page.tsx` — Guard `startCreating` during editing
- `components/routes/save-route-modal.tsx` — useEffect to sync existingData into form state

## Session 3 (Evening)

### Summary
Implemented the Route Variants feature — database migration, similarity detection algorithm, API endpoints, save-time detection banner, variant badges on route cards, Variants tab in route detail drawer, Create Route Variant fork flow, and explore map pin filtering.

### Changes

#### Feature: Route Variants — full implementation
The variants system links routes that share 65%+ geometry. The first route created is the "parent" (visible on explore map). Later similar routes become variants (hidden from explore pins but searchable, saveable, and viewable).

**Database migration** (`supabase/migrations/078_route_variants.sql`):
- `route_variants` table: bidirectional links between similar routes with similarity_score and source (auto/manual/fork)
- `show_on_explore` boolean column on routes (default true, set false for variants)
- `variant_of_id` UUID column on routes for direct parent tracking
- RLS policies: readable if either route is visible, insertable/deletable by route owners or admins

**Geometry similarity utility** (`lib/routes/geometry-similarity.ts`):
- `calculateGeometrySimilarity()` — samples points, checks bidirectional point-to-segment overlap within 50m tolerance
- `boundingBoxesOverlap()` — quick pre-filter to skip distant routes

**API endpoints**:
- `app/api/routes/[id]/variants/route.ts` — GET variants for a route, POST to create variant link
- `app/api/routes/check-similarity/route.ts` — POST geometry to find similar routes (65% threshold)

**Save-time detection** (`components/routes/save-route-modal.tsx`):
- Checks similarity when modal opens, shows amber banner: "This route shares X% with [Route Name]"
- Opt-out link: "Save as standalone instead"
- Blue banner for fork variants: "Variant of [Route Name]"

**Variant linking in save flow** (`app/routes/page.tsx`):
- After saving, creates variant link via POST and sets show_on_explore=false + variant_of_id via PATCH

**Variant badges** (multiple components):
- Purple "Variant" badge with GitBranch icon on route cards in Find Routes, Saved Routes, and RouteCard
- Variant count indicator on mini map cards (fetched from variants API)

**Variants tab in route detail drawer** (`components/routes/route-detail-drawer.tsx`):
- New tab between Warnings and Weather showing linked variants
- "Create Route Variant" button (purple, pencil icon)
- Variant cards with similarity percentage, distance, owner, and click-to-view

**Create Route Variant fork flow** (`app/routes/page.tsx`):
- `startForkingVariant()` — loads parent's spine_points into creation mode
- Sets variant metadata so save modal auto-links the new route

**Explore map pin filtering** (`app/routes/page.tsx`):
- Filters `show_on_explore !== false` routes from map pins

**API select query updates** (search, my-routes, favorites):
- Added `variant_of_id` and `show_on_explore` to all route select queries
