# 2026-04-10 — Homepage redesign + header refactor

## Session summary

Redesigned the homepage hero from a flat green gradient band into a contained-card photo hero, moved the main nav from the centre of the header to the top-right next to the avatar, fixed the "Become a Host" loading flicker for hosts and admins, swapped the logo to the new test logo, and reordered the homepage so Featured Properties sits directly beneath the hero.

## Files modified

- `app/page.tsx` — replaced green-gradient hero with photo hero (`/photo-hero.png`) inside a contained card with rounded corners, dark bottom gradient overlay, hero text + search bar overlaid. Reordered sections so Featured Properties now comes directly under the hero, with Why Choose padoq moved beneath it. Removed unused `Star` import.
- `components/header.tsx` — moved Search / Routes / Become a Host nav links from the centre into a flex group on the right next to the avatar / dashboard / hamburger. Logo source updated from `/logo.png` to `/logo-test.png`. Added `showBecomeHost` derived state that gates the "Become a Host" link on `!loading` so it never flashes for hosts and admins during the auth fetch.

## Files renamed

- `public/Test Logo.png` → `public/logo-test.png`
- `public/Photo Hero.png` → `public/photo-hero.png`

Filenames with spaces were causing awkward URL encoding (`%20`) and would cause friction in future code references. Renamed to kebab-case before any code referenced them.

## What changed and why

1. **Photo hero replaces green band** — the existing hero was a flat green gradient. The new design uses a real photograph of horses in a UK paddock, contained in a card with rounded corners (not full-bleed), matching the user-supplied reference image. A `bg-gradient-to-t from-black/70 via-black/20 to-transparent` overlay sits beneath the text to keep the white serif title and subhead legible against the bright photo.

2. **Header nav moved right** — Search / Routes / Become a Host were previously centred in the header (`flex-1 justify-center`). Wrapped in a flex group with the avatar / dashboard / hamburger so they all sit together in the top-right. The Padoq logo stays in the top-left.

3. **"Become a Host" flicker fix** — the previous gating expression `(!user || (user.role !== "host" && user.role !== "admin"))` evaluated to true while `user` was still `null` during the initial fetch, so hosts and admins briefly saw the link before it disappeared. Replaced with a `showBecomeHost` constant that requires `!loading`, so the link only ever renders once we know the user's role for sure. Logged-out visitors and guests still see it; hosts and admins never see it, even for a frame.

4. **Logo swap** — `/logo.png` → `/logo-test.png` per the new branding asset Richard dropped in `public/`.

5. **Section reorder** — Photo hero → Featured Properties → Why Choose padoq → Routes Teaser → Host CTA. Featured Properties moved up to sit immediately below the hero so the first thing visitors see after the search bar is real properties, not marketing tiles.

## Architectural decisions

- **Did not split Header into a server component + client island.** The original plan was to convert Header to an async server component that fetches the user via the server Supabase client, eliminating the flicker entirely. During implementation, grep revealed that 7 client-component pages import Header directly (`app/search/page.tsx`, `app/host/page.tsx`, `app/host/listings/page.tsx`, `app/routes/edit/[id]/page.tsx`, `app/account/edit/page.tsx`, `app/verify/complete/page.tsx`, `app/host/property/new/page.tsx`). Async components are not allowed inside client trees, so this would have required refactoring 7 unrelated pages to push their `"use client"` logic into child components. That violates CLAUDE.md's "no snuck-in refactors" rule and is out of scope for a homepage redesign. Switched to the loading-gated approach instead, which fixes the actual user-visible bug without dragging seven other files into the change.

## Risk notes

- **Photo hero asset is required.** `/photo-hero.png` must exist in `public/` or the homepage will render a broken image. Confirmed present after rename.
- **Logo swap is global.** Every page that uses `<Header />` now shows `logo-test.png`. If `logo-test.png` is a placeholder rather than the final brand asset, the user will see it everywhere, not just on the homepage.
- **Hero height is fixed at `h-[420px] md:h-[560px]`.** Tall enough to comfortably fit the title, subtitle, and search bar with room to breathe, but if the search bar grows in future (extra filters, etc) the bottom-aligned text could crowd the centre of the photo. Easy to bump if needed.
- **Skeleton-style fix means logged-out visitors still see "Become a Host" instantly** (correct), and hosts / admins see no link at all during the loading window (correct — they should never see it). Guests see it instantly too. The only behavioural change is hosts and admins no longer get the false-positive flash.

Hero top padding reduced from `pt-6 md:pt-10` to `pt-2 md:pt-3` so the hero sits flush under the nav bar.

### What changed and why

With 6 cards that would render as 1 full row + a half row, which looks unfinished, so the fetch limit was bumped to 8 for two clean rows. 7. **"Saved by X" line removed.** The line at `property-card.tsx:100-104` was rendering only when `favorite_count > 0`, which meant cards with saves were ~24px taller than cards without. In a multi-card grid this caused inconsistent row heights and bad alignment (Airbnb-style cards should always be the same shape). Removing the line gives every card a fixed height. The fetch in `app/page.tsx` is left in place per Richard's instruction.

Logo bump + slight left shift, plus a new editorial display font for headings.

- `components/header.tsx` — logo bumped from `width={160} height={56}` to `width={200} height={70}` (~25% larger). Added `-ml-3` (12px negative left margin) to the logo `<Link>` so the visible "P" character of the padoq mark sits at the photo hero's left edge instead of the PNG bounding box.

### What changed and why

1. **New display font for editorial headings.** "Featured Stays" was rendered in Playfair Display Bold, which is a chunky high-contrast Didone — visually heavy. Richard wanted something more refined and editorial. Added Cormorant Garamond, an old-style serif designed for display use, exposed as a new `font-display` Tailwind family. This is additive — the existing `font-serif` (Playfair) is still in place for everywhere else that uses it. Only the homepage Featured Stays heading uses `font-display` for now. If Richard likes it, easy to roll out to other headings later.
2. **Logo 25% bigger.** From 160x56 to 200x70. The morning's bump from 120x40 to 160x56 was a step in the right direction but Richard felt it still wasn't dominant enough next to the new larger photo hero. 200x70 fills the 80px header height (`h-20`) more confidently while still having a small breathing margin top and bottom.
3. **Negative left margin on the logo.** PNG logos always have some internal whitespace around the actual glyph. The previous version had the PNG's left edge sitting at the wrapper's `px-4` boundary, which meant the visible "P" character was offset further inward. `-ml-3` (12px) shifts the PNG left so the visible "P" lines up with the photo hero card's left edge, matching Richard's "the P should be inline with the edge of the photo hero" ask. The exact value (12px) is a guess based on typical logo whitespace; may need a small tweak depending on how the rendered logo looks in browser.

### Risk notes

- **Cormorant Garamond is a new HTTP request** during initial page load. Next.js's `next/font` system self-hosts the font and inlines `font-display: swap`, so there's no FOIT (flash of invisible text) — text renders immediately in the fallback (Georgia / generic serif) and swaps in once the font loads. Performance hit is minimal but real.
- **Negative left margin can cause overflow.** `-ml-3` on the logo `<Link>` shifts it 12px outside the wrapper's left padding. Since the wrapper is `px-2 md:px-4` (8px / 16px), the shift takes the logo to either +4px from the viewport edge (mobile) or -4px past it (desktop). In practice this is fine because the photo hero below also uses `px-2 md:px-4`, but on very narrow viewports (<320px) the logo could clip the screen edge. Modern phones are 360px+ so this is theoretical.
- **The new font may not match what Richard pictured.** The reference image of "Path Erosion" is from a different context I couldn't fully trace in the codebase. Cormorant Garamond is my best guess at a refined editorial serif that matches the visual character. If it doesn't match, the swap is one className change away — alternatives to try in order would be EB Garamond, Crimson Text, Lora, or Playfair Display at regular weight.

## Search button height (afternoon, fifth round)

### Files modified

- `components/search-bar.tsx` — search button column collapsed. The previous `flex flex-col` wrapper plus invisible label spacer plus `h-full` button was being defeated by `size="lg"` baking in a fixed `h-11` (44px) on the button itself. Replaced the whole column with a direct `<Button>` child of the grid: dropped `size="lg"`, set explicit `h-[88px]` (double the previous 44px), kept the larger-button feel via `text-base px-8`, and added `self-end` so the button sits at the bottom of the grid row (matching the bottom of the labelled inputs in the other columns).

### What changed and why

The morning's invisible-label-plus-`h-full` trick was supposed to make the search button stretch to the full column height. It looked right in theory but didn't actually work in browser because shadcn's Button component has size variants that bake in fixed heights via class-variance-authority — `size="lg"` adds `h-11` to the button's class string. `cn()` uses `tailwind-merge` so a later class can override an earlier one, but `h-full` and `h-11` were both being applied with similar specificity and the merge wasn't deduplicating them the way I expected, so the button stayed at h-11.

The fix is to drop `size="lg"` entirely (so no fixed height is set) and put the desired height on the className directly. `h-[88px]` is exactly double the previous 44px, which is what Richard literally asked for. With the button now driving its own height, the form's grid row grows to fit, and the entire white form box gets taller — the goal is "fill the outer box a bit more" which this achieves by making the button big enough that the box no longer has empty space inside it.

`self-end` keeps the button visually anchored to the bottom of the row, in line with the input fields in the other columns (which sit at the bottom of their cells under their labels).

### Risk notes

- **The form is now visibly taller.** Adding ~44px of button height grows the whole form box by ~44px because the grid row stretches to fit the button. This is the intended outcome but worth flagging — if it ends up looking too chunky, easy to back off to `h-16` (64px) or `h-20` (80px).
- **The other columns now have empty space at the top.** With the row at 88px and label+input at ~68px, there's ~20px of extra space in the labelled columns. The labels and inputs sit at the bottom of their cells (default grid alignment), so the empty space is at the top, which looks fine.

### Verification

1. Search button on the homepage is visibly much taller — roughly twice the height it was before.
2. The white form box around the search bar is taller to accommodate the button.
3. Button is bottom-aligned with the input fields in the other columns.
4. `npm run build` clean.
