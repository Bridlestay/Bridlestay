# Changelog — 2026-03-13

## Session 1: Fix route edit saving as new route + no-photo rounded corners

### Summary
Fixed two issues: (1) Editing a route created a new route instead of updating because the SaveRouteModal kept stale form state between uses, and (2) the "No photos yet" placeholder in the route detail modal had no visible rounded corners.

### Files Modified
- `app/routes/page.tsx` — Added `key` prop to `SaveRouteModal` to force remount between create/edit modes
- `components/routes/route-detail-drawer.tsx` — Inset the "No photos yet" placeholder with padding and rounded corners

### What Changed and Why

**Route edit creating new routes (Issue 1)**:
- **Root cause**: The `SaveRouteModal` component uses `useState` to initialize form fields from `existingData` props. However, `useState` initializers only run on first mount. Since the modal stays mounted (returns `null` when closed rather than being removed from the tree), switching between create and edit modes left stale form state — empty fields, wrong button text, and incorrect save behavior.
- **Fix**: Added a `key` prop (`key={isEditing ? 'edit-${editingRouteId}' : 'create'}`) to the `SaveRouteModal`. When the key changes (switching between create/edit or editing a different route), React unmounts and remounts the component, ensuring `useState` initializers run fresh with the correct `existingData`. This means the form pre-populates with existing title, description, visibility, and difficulty when editing, and the button correctly shows "Update Route".

**No-photo rounded corners (Issue 2)**:
- **Root cause**: The nested `overflow-y-auto` scrollable div inside the card prevented the card's `rounded-2xl overflow-hidden` from visually clipping the green gradient at the top corners.
- **Fix**: Added `rounded-t-2xl` directly to the "No photos yet" gradient div so it matches the card's corner radius. The gradient stays edge-to-edge (same layout as the photo carousel) but now has properly rounded top corners.

### Verification Steps
1. `npm run build` passes
2. **Edit existing route**: Open route detail → three-dot menu → Edit → modify route → Save Route → modal shows "Update Route" button with pre-filled title/description → saves correctly as update (no new route created)
3. **Create new route**: Start fresh route → Save Route → modal shows "Save Route" button with empty fields → creates new route
4. **No-photo styling**: Open a route with no photos → white content area below the green gradient has rounded top corners overlapping the hero, same as routes with photos
5. **Photo styling**: Routes with photos still show edge-to-edge photo carousel (no change)

## Session 2: Fix no-photo rounded corners (correct approach) + deterministic undo

### Summary
Fixed the rounded corners on the "No photos yet" route detail card (the white content area below the hero now has the same `rounded-t-2xl -mt-4` pull-up effect as routes with photos). Also made undo deterministic by saving snapped segments in the undo history — undo now restores the exact previous route geometry instead of re-fetching from Mapbox.

### Files Modified
- `components/routes/route-detail-drawer.tsx` — Apply `rounded-t-2xl -mt-4` to content area for all routes (not just photos)
- `components/routes/routes-map-mapbox.tsx` — Added `getSnappedSegments` and `restoreSnappedSegments` methods to map ref
- `app/routes/page.tsx` — Changed undo history to store `{ waypoints, segments }` snapshots; undo restores both directly

### What Changed and Why

**No-photo rounded corners (corrected)**:
- **Root cause**: The content area below the photo hero (`<div className="relative z-10 bg-white">`) only got `rounded-t-2xl -mt-4` when photos were present (`displayPhotosForCarousel.length > 0`). Without photos, the content had no rounded top corners and no negative margin overlap.
- **Fix**: Apply `rounded-t-2xl -mt-4` and `pt-6` unconditionally to the content area, so both photo and no-photo routes have the same card pull-up visual.

**Deterministic undo with snapped segments**:
- **Root cause**: Undo cleared all snapped segments and triggered a full re-fetch from Mapbox Directions API. The re-fetched segments could produce different road-following geometry than what the user had before, causing the route to "jank" to different roads.
- **Fix**: The undo history now stores a deep copy of the snapped segments alongside waypoints. On undo, both are restored directly — no Mapbox API calls, instant, and the route looks exactly as it did before the last change. This works correctly regardless of whether snap was on or off at the time, or whether the user has since toggled snap mode.

### Risk Notes
- History entries are slightly larger (include segment coordinate arrays). For a typical route with 20 undo steps, this adds ~50-100KB — negligible.
- `getSnappedSegments()` returns a deep copy so mutations to the live segments don't affect stored history.

### Verification Steps
1. `npm run build` passes
2. **No-photo rounded corners**: Open a route with no photos → white content area has rounded top corners overlapping the green gradient, matching the photo layout
3. **Undo with snap on**: Place 4+ waypoints with snap enabled → undo → route instantly reverts to exact previous geometry (no re-fetch, no janking to different roads)
4. **Undo across snap toggle**: Place points with snap on → toggle snap off → undo → route restores with snapped geometry from when it was snapped
5. **Multiple undo**: Undo several times in succession → each step restores the exact previous state
6. **Undo after erase**: Erase a waypoint → undo → route restores with correct segments
