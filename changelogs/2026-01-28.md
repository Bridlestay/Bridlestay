## 1. Fixed Magic Links & Password Reset URLs (localhost → Production)

**Problem:** Magic links and password reset emails were redirecting to `localhost:3000` instead of the live website URL.

**Root Cause:** The `app/api/admin/user-actions/route.ts` was using `process.env.NEXT_PUBLIC_APP_URL` which might not be set in production, falling back to `localhost:3000`.

**Solution:** 
- Added intelligent URL detection using request headers (`host` and `x-forwarded-proto`)
- Falls back to `https://bridlestay.vercel.app` if no headers available
- Fixed password reset redirect to go to `/auth/update-password` (not `/auth/reset-password`)

**Files Modified:**
- `app/api/admin/user-actions/route.ts` - Added `getAppUrl()` helper function

**How it works:**
1. First checks `NEXT_PUBLIC_APP_URL` environment variable
2. Then reads request headers for host and protocol
3. Constructs proper URL dynamically for each request

## 6. Documentation: Admin Action Security & Clarity

### Impersonation Security Model

**Q: Is "Login as User" legal and secure?**

**A: Yes, when implemented correctly:**

1. **Audit Logging** - Every impersonation is logged to `admin_audit_log` with:
   - Admin ID and name
   - Target user ID and email
   - Timestamp
   - Details of the action

2. **Who Can Do It** - Only users with `role = 'admin'` in the database

3. **How It Works:**
   - Admin generates a magic link for the target user
   - Magic link is routed through `/auth/impersonate` page
   - Admin is signed out first, then authenticated as target user
   - This creates a proper session for the target user

4. **Legal Considerations:**
   - Most platforms include impersonation in Terms of Service
   - Audit trail provides accountability
   - Should only be used for support/debugging purposes

### Ban/Warn/Softban Clarity

**Coming Soon:** The warning/banning system needs clearer UI to explain:
- What exactly happens when each action is taken
- Whether the user can still access inbox, make bookings, receive payouts
- Duration of restrictions
- How the user is notified

**Recommendation:** Add a "What does this do?" info tooltip next to each action button that explains:
- **Warn**: Sends educational warning to user's inbox, recorded in history, trust score may decrease
- **Softban**: Restricts posting/messaging but allows reading, booking, and payouts
- **Ban**: Full account suspension, no actions allowed

## Database Migration: 063_admin_panel_improvements.sql

Run this migration in Supabase SQL Editor:

```sql
-- 1. Feedback important flag
ALTER TABLE user_feedback ADD COLUMN IF NOT EXISTS is_important BOOLEAN DEFAULT false;
CREATE INDEX IF NOT EXISTS idx_user_feedback_important ON user_feedback(is_important) WHERE is_important = true;

-- 2. User referral fee discounts
ALTER TABLE users
ADD COLUMN IF NOT EXISTS referral_fee_discount_percent INTEGER DEFAULT 0,
ADD COLUMN IF NOT EXISTS referral_discount_bookings_remaining INTEGER DEFAULT 0;

-- 3. Referral rewards tracking table
CREATE TABLE IF NOT EXISTS user_referral_rewards (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  reward_type TEXT NOT NULL CHECK (reward_type IN ('fee_discount', 'badge')),
  discount_percent INTEGER,
  bookings_applicable INTEGER DEFAULT 1,
  bookings_used INTEGER DEFAULT 0,
  from_referral_id UUID REFERENCES referral_redemptions(id),
  status TEXT DEFAULT 'active' CHECK (status IN ('active', 'used', 'expired')),
  expires_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE user_referral_rewards ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can view their own rewards" ON user_referral_rewards FOR SELECT USING (user_id = auth.uid());
CREATE POLICY "Admins can manage all rewards" ON user_referral_rewards FOR ALL USING (EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND role = 'admin'));
```
