## Session 4: Await notification calls for Vercel serverless reliability

### Summary
Added `await` to all fire-and-forget `createNotification()` and `createNotifications()` calls across 13 API routes. On Vercel serverless, the function execution can terminate before unawaited promises complete, causing notifications to silently fail.

### Files Modified
- `app/api/messages/send/route.ts` — `await createNotification()`
- `app/api/routes/[id]/waypoint-suggestions/route.ts` — `await createNotification()`
- `app/api/routes/[id]/waypoint-suggestions/[suggestionId]/route.ts` — `await createNotification()` (2 calls: approved + rejected)
- `app/api/routes/[id]/like/route.ts` — `await createNotification()`
- `app/api/routes/[id]/comments/route.ts` — `await createNotification()` (2 calls: reply + route owner)
- `app/api/routes/[id]/hazards/route.ts` — `await createNotification()`
- `app/api/routes/[id]/review/route.ts` — `await createNotification()`
- `app/api/reviews/route.ts` — `await createNotification()`
- `app/api/booking/request/route.ts` — `await createNotification()`
- `app/api/booking/accept/route.ts` — `await createNotification()`
- `app/api/booking/decline/route.ts` — `await createNotification()`
- `app/api/booking/cancel/route.ts` — `await createNotification()`
- `app/api/host/become/route.ts` — `await createNotifications()`

### What Changed and Why
All 15 notification calls were fire-and-forget (no `await`). On Vercel serverless, the function execution context is torn down once the response is sent, so unawaited promises may never complete. Adding `await` ensures notifications are inserted into the database before the API response is returned.

### Risk Notes
- Adds a small amount of latency to each API response (one DB insert). The `createNotification` helper already has try/catch and silently fails, so notification errors will not break primary operations.

## Session 5: Fix notifications API 500 error

### Summary
Fixed the `/api/notifications` GET endpoint returning 500 Internal Server Error. The PostgREST FK join `users!notifications_actor_id_fkey` failed because the `actor_id` FK references `auth.users` (auth schema) but the join attempted to query `public.users`. Replaced with a two-step fetch: query notifications first, then batch-fetch actor details from `public.users` separately.

### Files Modified
- `app/api/notifications/route.ts` — Removed broken FK join from `.select()`, added separate actor lookup by collecting unique `actor_id` values and fetching from `public.users`, then attaching actor data to each notification before returning

### What Changed and Why
The original query used `.select("*, actor:users!notifications_actor_id_fkey(id, name, avatar_url)")` which relies on PostgREST resolving the FK relationship. However, the `notifications.actor_id` FK points to `auth.users(id)` (the Supabase auth schema), not `public.users`. PostgREST cannot bridge across schemas with FK hints, causing a 500 error. The fix fetches notifications without any join, then does a second query to `public.users` for actor details (name, avatar_url) and merges them client-side.

### Risk Notes
- Adds one extra DB query per request (batch actor lookup), but only when notifications have actors — negligible overhead
- If a user exists in `auth.users` but not `public.users`, their actor data will be null (gracefully handled)

## Session 8: Fix route plotting — stale closure in map click handler

### Summary
Fixed route creation plotting not working. The Mapbox map click handler was registered once on mount inside a `useEffect([isLoaded])`, capturing `isCreating`, `isPlotting`, `toolMode`, `onWaypointAdd`, and `waypoints` as stale closures. When these values changed later (e.g., user enters creation mode), the handler still saw the initial `false`/`undefined` values and silently returned.

### Files Modified
- `components/routes/routes-map-mapbox.tsx` — Added refs (`isCreatingRef`, `isPlottingRef`, `toolModeRef`, `onWaypointAddRef`, `waypointsRef`) that are updated on every render. The map click handler now reads from these refs instead of the stale closure variables.

### What Changed and Why
The map click handler at line ~383 was inside a `useEffect` with dependency `[isLoaded]`, meaning it ran once when Mapbox loaded. All variables inside the closure (`isCreating`, `isPlotting`, etc.) were captured at that moment — typically `false`/`undefined`. When the user later entered creation mode, the handler still checked the original values and returned early. The fix uses refs (which are mutable objects) so the closure always reads the current value.

### Risk Notes
- This was a pre-existing bug that was masked because the old side panel layout caused the map to re-mount when entering creation mode. The new full-screen map persists across mode changes, exposing the stale closure.

## Session 9: Fix all route creation tools — drag, erase, insert, snap, circular detection

### Summary
Fixed broken waypoint dragging and erasing (function signature mismatches between map component and page callbacks), and implemented three previously stubbed features: insert mode, snap-to-path, and circular route detection.

### Files Modified
- `components/routes/routes-map-mapbox.tsx` — Fixed prop types, added callback refs, implemented insert/snap/circular features

### What Changed and Why

1. **Fixed waypoint drag (broken)**: The map called `onWaypointUpdate(index, waypointObj)` but the page expected `(id, lat, lng, snapped?)`. Dragging silently failed because `index` (number) was passed as `id` (string). Now calls `onWaypointUpdateRef.current(wp.id, lngLat.lat, lngLat.lng)`.

2. **Fixed waypoint erase (broken)**: The map called `onWaypointRemove(index)` (number) but the page expected `(id)` (string). Now calls `onWaypointRemoveRef.current(wp.id)`.

3. **Added callback refs**: Added refs for `onWaypointUpdate`, `onWaypointRemove`, `onWaypointInsert`, `onCircularDetected`, and `snapEnabled` — updated on every render to avoid stale closures in map event handlers.

4. **Implemented insert mode**: When `toolMode === "insert"`, clicking on the route line inserts a new waypoint at the clicked position between the two nearest existing waypoints. Uses a transparent 20px-wide hit area layer for easier clicking. Cursor changes to "copy" on hover.

5. **Implemented snap-to-path**: When `snapEnabled` is true, new waypoints are snapped to the nearest road/path using the Mapbox Map Matching API (`/matching/v5/mapbox/walking/`). Falls back to unsnapped placement if the API call fails or no match is found.

6. **Implemented circular route detection**: After adding waypoints, if the last waypoint is within 50m of the first (3+ waypoints), `onCircularDetected` fires to close the route into a loop.

7. **Fixed cursor per tool mode**: Waypoint markers now show `pointer` cursor in erase mode and `move` cursor in plot mode.

8. **Updated prop types**: `onWaypointUpdate`, `onWaypointRemove`, and `onWaypointInsert` signatures now match what the page provides. Removed "Not yet implemented" comments.

### Risk Notes
- Snap-to-path uses the Mapbox Map Matching API which counts against the Mapbox API quota (free tier: 100k requests/month)
- Insert mode uses `pointToSegmentDistance` in coordinate space (not geodesic) — accurate enough for UI-level segment detection
- Circular detection threshold is 50m — may need tuning based on user feedback
