# Changelog — 2026-03-08

## Session 1: Waypoint photo viewing and navigation fixes

### Summary
Added photo carousel navigation and lightbox viewing to waypoint editor and waypoint cards. Fixed suggestion approval not immediately displaying newly approved photos.

### Files Modified
- `components/routes/edit-waypoint-view.tsx` — Added photo carousel arrows in the header banner, clickable photo thumbnails, PhotoLightbox integration, safe index clamping, and refetch-based photo sync after suggestion approval
- `components/routes/waypoint-card.tsx` — Made all photo collage layouts clickable to open a full-screen PhotoLightbox viewer

### What Changed and Why
1. **Editor header photo navigation**: Previously only showed the first photo with no way to browse. Added left/right chevron arrows to cycle through photos, and clicking the photo opens a full-screen lightbox.
2. **Editor photo grid**: Thumbnail photos in the grid are now clickable — opens lightbox at the clicked photo index.
3. **Waypoint card photo collage**: All photo layouts (single, two-up, 3+ collage) are now clickable buttons that open the existing PhotoLightbox component for full-screen viewing with navigation.
4. **Suggestion approval photo display**: Previously used `crypto.randomUUID()` to create temporary photo objects client-side after approval. Replaced with a `refetchPhotos()` call that re-fetches actual DB records, ensuring photos display correctly with real IDs. Also added this refetch to the "Accept photos only" handler which was missing photo state updates entirely.
5. **Safe index clamping**: Added `safePhotoIndex` to prevent out-of-bounds errors if photos are deleted while viewing.

### Risk Notes
- PhotoLightbox uses `z-[100]` which should layer above all other UI elements
- The lightbox in waypoint cards renders inside each card — if many cards are on screen, only the active one will show its lightbox

### Verification Steps
1. Open a waypoint with multiple photos in the editor — verify left/right arrows cycle through photos
2. Click a photo in the editor header — verify full-screen lightbox opens with navigation
3. Click a photo thumbnail in the editor's Photos grid — verify lightbox opens at that photo
4. On the route detail page, click a photo in a waypoint card — verify lightbox opens
5. Approve a suggestion with photos — verify photos appear in the editor gallery immediately
6. Use "Accept photos only" — verify photos appear in the editor gallery immediately

## Session 2: Unified photo collection in route hero carousel

### Summary
All photos across the entire route — route display photos, waypoint photos, and review/completion photos — are now collected into a single photo hero carousel at the top of the route detail card. The carousel auto-scrolls every 6 seconds and pauses on user interaction.

### Files Modified
- `components/routes/route-detail-drawer.tsx` — Merged all photo sources (display, non-display route photos, community/review `userPhotos`, waypoint photos) into `displayPhotosForCarousel`; added auto-scroll interval with pause-on-interaction

### What Changed and Why
1. **Waypoint photos in hero carousel**: All waypoint photos (primary `photo_url` + community `waypoint_photos`) are collected into a flat array and appended after route photos in the carousel.
2. **Review/community photos in hero carousel**: `userPhotos` (from `route_user_photos` — includes review uploads) are now included in the carousel. Previously only display-flagged route photos appeared; community/review photos were only in the gallery. The initial dedup approach failed because completion photos share the same IDs as `userPhotos` — the fix includes `userPhotos` directly in the carousel instead.
3. **Non-display route photos**: Route photos not flagged as `is_display` are also now included after display photos, so all owner photos appear.
4. **Gallery and count updated**: The full photo gallery and "See all X" count badge include waypoint photos.
5. **Auto-scroll**: The carousel auto-advances every 6 seconds. Clicking arrows, dot indicators, or opening the lightbox pauses auto-scroll for 6 seconds before resuming.

### Risk Notes
- Dot indicators hidden when total photos > 8 (existing behaviour) — with all photos collected this is more likely to trigger

### Verification Steps
1. Open a route with waypoint photos — verify they appear in the hero carousel after route photos
2. Open a route with review/completion photos — verify they also appear in the carousel
3. Verify the carousel auto-scrolls every ~6 seconds
4. Click an arrow or dot — verify auto-scroll pauses then resumes
5. Check "See all X" count includes all photo sources
6. Open the full photo gallery — verify waypoint and review photos appear

## Session 3: Full notification system

### Summary
Built a complete in-app notification system: database schema, API routes, notifications page UI, header integration with unread dot indicators, and notification hooks into all existing API flows (messages, bookings, suggestions, comments, reviews, likes, hazards, host applications).

### Files Created
- `supabase/migrations/076_notifications.sql` — Notifications table with indexes and RLS policies
- `lib/notifications.ts` — `createNotification()` and `createNotifications()` helper functions (service client, silent fail, self-notification skip)
- `app/api/notifications/route.ts` — GET (paginated, filterable) + PATCH (mark read / mark all read)
- `app/api/notifications/unread-count/route.ts` — GET unread count for header badge
- `app/notifications/page.tsx` — Full notifications feed page with All/Unread filter tabs, mark all read, notification items with actor avatars, type icons, relative timestamps, unread indicators

### Files Modified
- `components/header.tsx` — Added Bell icon, `notifCount` state with 30s polling, red dot on burger menu when unread, Notifications menu item with count badge
- `app/api/messages/send/route.ts` — Added `message` notification to recipient
- `app/api/routes/[id]/waypoint-suggestions/route.ts` — Added `suggestion_received` notification to route owner
- `app/api/routes/[id]/waypoint-suggestions/[suggestionId]/route.ts` — Added `suggestion_approved`/`suggestion_rejected` notifications to suggester
- `app/api/routes/[id]/like/route.ts` — Added `route_liked` notification to route owner
- `app/api/routes/[id]/comments/route.ts` — Added `route_comment` notification to route owner + parent commenter for replies
- `app/api/routes/[id]/hazards/route.ts` — Added `hazard_reported` notification to route owner
- `app/api/routes/[id]/review/route.ts` — Added `route_review` notification to route owner
- `app/api/reviews/route.ts` — Added `listing_review` notification to host
- `app/api/booking/request/route.ts` — Added `booking_request` notification to host
- `app/api/booking/accept/route.ts` — Added `booking_accepted` notification to guest
- `app/api/booking/decline/route.ts` — Added `booking_declined` notification to guest
- `app/api/booking/cancel/route.ts` — Added `booking_cancelled` notification to other party
- `app/api/host/become/route.ts` — Added `host_application` notification to all admins

### What Changed and Why
1. **Database**: New `notifications` table with user_id, type, title, body, link, actor_id, read flag. Optimized indexes for unread queries. RLS policies: users read/update own, service role inserts.
2. **Helper library**: `createNotification()` uses service client to bypass RLS, silently fails to never block primary operations, skips self-notifications. `createNotifications()` for batch inserts (used by host application → all admins).
3. **API**: GET endpoint with pagination, unread filter, actor join. PATCH for marking individual or all as read. Separate unread-count endpoint for lightweight header polling.
4. **Notifications page**: Full feed UI with icon/color mapping for all 22 notification types, actor avatars, relative timestamps, click-to-navigate with auto mark-as-read, empty states.
5. **Header**: Red dot indicator on burger menu icon when unread notifications or messages exist. Notifications menu item shows count + red dot.
6. **Hooks**: All major user actions now generate notifications — messages, suggestions (create/approve/reject), route interactions (like, comment, review, hazard), bookings (request/accept/decline/cancel), listing reviews, and host applications.

### Notification Types (22 total)
message, message_reply, route_comment, route_liked, route_completed, route_review, suggestion_received, suggestion_photo, suggestion_approved, suggestion_rejected, hazard_reported, booking_request, booking_accepted, booking_declined, booking_cancelled, payment_received, refund_processed, listing_review, listing_approved, report_filed, host_application, damage_claim

### SQL Migration Required
Run `supabase/migrations/076_notifications.sql` in Supabase SQL editor.

### Risk Notes
- Notification creation is fire-and-forget (no await) — if it fails, the primary action still succeeds
- Polling at 30s intervals for unread count — acceptable for current scale
- Self-notification skip prevents users from seeing notifications about their own actions
