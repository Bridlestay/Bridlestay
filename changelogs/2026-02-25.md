# Changelog — 2026-02-25

## Session: Rider Reviews Overhaul — Data Fix + UI Redesign

Fixed all three root causes preventing rider reviews from displaying properly (text, tags, photos) and redesigned the review cards to follow the Padoq UI/UX design guide.

---

### Root Cause Analysis

The review system had three separate issues:

1. **`review_text` column query failure**: The completions API queried `route_reviews.review_text`, but if migration 069 hadn't been run, the query silently failed and `review_body` was always empty. Users saw only the short note (or nothing).

2. **Photo matching broken for route owners**: Owner review photos were stored in `route_photos` (with `uploaded_by_user_id`), but the display code only checked `route_user_photos` (with `user_id`). Owner review photos were invisible.

3. **Duplicate completion race condition** (fixed in prior session): `/review` API created a bare completion (no tags/notes) before `/complete` could store them. Call order was swapped but existing data remained stale.

---

### Fix 1: Resilient Completions API

**File:** `app/api/routes/[id]/completions/route.ts`

The API now handles all data in a single response:

- **Review text**: Tries `select("user_id, review_text, body")` first. If the query fails (column doesn't exist), falls back to `select("user_id, body")`. Either way, builds the `reviewBodyMap` from whichever field has data.
- **Photos**: Queries BOTH `route_user_photos` (community photos, matched by `user_id`) and `route_photos` (owner photos, matched by `uploaded_by_user_id`). Returns a `photos` array per completion.
- **Tags/notes**: Unchanged — parsed from `route_completions.notes` JSON.

Each completion in the response now includes `{ id, user, completed_at, tags, short_note, review_body, photos }`.

### Fix 2: Review Card Redesign

**File:** `components/routes/route-detail-drawer.tsx`

Redesigned following the UI/UX Design Guide:

- **Card style**: `bg-white border border-gray-200 rounded-xl p-5` — warm, Airbnb-inspired cards with generous padding
- **Avatar**: Larger (40px), green-50 fallback background, semibold initials
- **Review text**: Shows `review_body` (full text from route_reviews) with fallback to `short_note`. Truncates at 200 chars with "Read more" toggle.
- **Tags**: Pill-shaped spans (`rounded-full bg-gray-100 text-gray-700`) with emoji + label
- **Photo grid**: Responsive — single photo gets `aspect-video` (wide), 2 photos get `grid-cols-2`, 3+ get `grid-cols-3`. Smooth hover scale effect. Click opens lightbox.
- **Photos from API**: Now uses `completion.photos` array from the API response instead of trying to match `userPhotos` by user_id. This is reliable regardless of which table the photo lives in.
- **Section header**: Clean "Rider Reviews" with count text instead of badge
- **Show all button**: `variant="outline" rounded-full` per button hierarchy guidelines
- **Long note refresh**: `handleSubmitLongNote` now calls `fetchCompletions()` after saving so the updated text appears immediately

### Risk Notes

- The completions API now makes 3-4 queries instead of 2 (completions + reviews + user_photos + route_photos). All are simple indexed queries on route_id, so performance impact is minimal.
- `route_photos` results include ALL photos for the route (including cover/display). This means owner's official route photos may appear under their review. This is acceptable — they uploaded them for this route.
- Existing reviews with bare completions (no tags) will still show the review text from `route_reviews`, just without tags. Tags will appear on new reviews submitted after the call-order fix.

### Verification

4. Upload a photo during review (as non-owner) → photo appears in the review card grid
