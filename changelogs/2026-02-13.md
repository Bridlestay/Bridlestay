# Changelog — 2026-02-13

## Photo Upload, Reviews Display & Completions Count

### Session Summary

Implemented real photo upload in the review flow (Step 2), added a reviews display section in the route detail modal, and ensured completions count updates correctly when riders complete a route.

### Files Created

- `app/api/routes/[id]/completions/route.ts` — New GET endpoint returning completions with user info and parsed tags/notes
- `supabase/migrations/066_completions_count_trigger.sql` — Trigger to auto-increment/decrement `completions_count` on routes table + backfill

### Files Modified

- `components/routes/route-detail-drawer.tsx`
  - **Step 2 photo upload**: Replaced "Coming soon" placeholder with working upload — user selects a category tag (View, Feature, Seasonal, Horse, Hazard, Parking), picks a photo, sees preview, uploads to Supabase storage
  - **Photo tag badges**: Carousel shows category badge in top-left corner for tagged photos
  - **Reviews section**: New "Rider Reviews" section between Waypoints/Hazards and Engagement Stats — shows user avatar, name, time ago, review tags as pills, and short notes. First 3 shown with "Show all" button
  - **Fetch completions**: Added `fetchCompletions()` call on mount to load review data
  - Added state for photo upload (file, preview, category) and completions data
  - Added `validateRoutePhotoUpload` import and file input ref

- `app/api/routes/[id]/review/route.ts`
  - After inserting a completion, recounts all completions and updates `completions_count` on the routes table

- `app/api/routes/[id]/complete/route.ts`
  - Same completions count update after successful insert

- `app/globals.css` — (no new changes this session, scrollbar-hidden from earlier)

### What Changed and Why

1. **Photo upload** was showing "Coming soon" — riders can now contribute tagged community photos during the review flow. Tags help categorize photos (View, Hazard, Parking, etc.) and show as badges in the carousel.

2. **Reviews had no display** — the submission flow worked but reviews were never shown. Now completions with tags and notes display in a card-based section visible to all users.

3. **Completions count stayed at 0** — neither the review nor complete API handlers were incrementing the count. Added explicit count updates in both handlers, plus a SQL trigger for long-term reliability.

### Risk Notes

- SQL migration 066 must be run in Supabase dashboard for the trigger to take effect. Without it, the API handlers still update the count manually.
- Photo uploads go to the `route_user_photos` table for non-owner users (community photos). The existing photo API handles this correctly.
- The completions API endpoint parses JSON from the `notes` field — if notes contain non-JSON text, it falls back to treating it as plain text.


### Verification Steps

1. Open route modal → "I've ridden this route!" → Step 2: select a category tag → tap upload area → pick image → see preview → "Upload & Next"
2. Complete full review flow (rating, photo, tags, note) → completions count should increment
3. Close and reopen modal → "Rider Reviews" section should show your review with tags and note
4. Photo carousel should show tag badge (e.g. "🏞️ View") in top-left corner for tagged photos

## Session 2: Elevation Profile, Waypoints Redesign & Map Waypoint Markers

### Session Summary

Added real elevation data via open-meteo.com API, redesigned the waypoints panel with expandable accordion items (matching OS Maps style), added waypoint markers to the Mapbox map, and enabled click-to-navigate from waypoints to map positions.

### Files Created

- `app/api/routes/[id]/elevation/route.ts` — GET endpoint that fetches elevation data from open-meteo.com, samples up to 200 points from route geometry, returns elevation profile + per-waypoint elevations, caches `elevation_gain_m`/`elevation_loss_m` on the routes table
- `lib/routes/os-grid-ref.ts` — WGS84 → UK Ordnance Survey Grid Reference conversion utility (Helmert transform + Transverse Mercator projection)

### Files Modified

- `components/routes/elevation-profile.tsx`
  - Rewritten to accept pre-computed `elevations[]` and `distances[]` from API
  - Purple/magenta gradient fill, proper Y-axis labels in metres, X-axis labels in km
  - Falls back to legacy coordinate-based mode if pre-computed data not provided

- `components/routes/route-detail-drawer.tsx`
  - **Elevation integration**: Added elevation state, `fetchElevation()` function, useEffect to fetch on route load
  - **Stats grid**: Ascent/Descent now show real values from elevation API (with "..." while loading)
  - **Elevation profile**: Added ElevationProfile component between stats grid and description
  - **Waypoints redesign**: Complete rewrite of waypoints panel — expandable accordion items with Start (green), numbered waypoints, Finish (red). Expanded view shows: description, distance from previous, ascent/descent from previous, OS Grid Ref, lat/long, elevation, "Show on map" button
  - **Compact waypoints card**: Updated preview to show Start → X points → Finish with colored dots
  - Added `onFlyToLocation` prop for map navigation from waypoint clicks

- `components/routes/routes-map-mapbox.tsx`
  - Added `routeWaypoints` and `showWaypoints` props
  - Added useEffect to render blue numbered circular markers at waypoint positions with name/description popups

- `app/routes/page.tsx`
  - Added `selectedRouteWaypoints` state and `fetchRouteWaypoints()` function
  - Pass waypoint data + toggle to map component
  - Wire up `onFlyToLocation` callback to `mapRef.current?.flyTo(lat, lng, 16)`

- `lib/routes/gpx-converter.ts`
  - `parseGPX()`: Now reads `<ele>` tags from track/route points, produces 3D coordinates
  - `convertToGPX()`: Writes `<ele>` tags when 3D coordinates present
  - `validateGeometry()`: Updated to accept coordinates with length 2 or 3

- `lib/routes/kml-parser.ts`
  - `parseCoordinates()`: Now extracts altitude (3rd value) from KML `lng,lat,alt` format

- `components/routes/map-layer-controls.tsx`
  - Renamed "Waymarkers" label to "Waypoints"

### What Changed and Why

1. **Elevation data was missing** — Routes showed "—" for Ascent/Descent. Created an API endpoint using the free open-meteo.com elevation API that looks up elevation for sampled route points and caches results in the DB.

2. **Waypoints were a flat list** — Redesigned to match OS Maps style with expandable accordion items showing description, distances, elevation, OS Grid Reference, and coordinates. Start/Finish are auto-generated from route geometry.

3. **No waypoint markers on map** — The "Waymarkers" toggle existed but did nothing. Added actual marker rendering at waypoint positions with numbered circles and popups.

4. **No click-to-navigate** — Added `onFlyToLocation` callback so clicking "Show on map" in a waypoint flies the map to that position.

5. **GPX/KML parsers discarded elevation** — Both parsers now preserve 3D coordinate data when available.

### Risk Notes

- The open-meteo.com API is free and has no API key, but may have rate limits for very high traffic
- Elevation data is cached after first lookup (writes to `elevation_gain_m`/`elevation_loss_m` columns)
- OS Grid Reference conversion only works for UK coordinates (returns null for locations outside UK bounds)

### Verification Steps

1. Open a route → elevation profile graph appears below stats grid with real data
2. Stats grid shows actual Ascent/Descent values (not "—")
3. Waypoints panel shows Start → named waypoints → Finish with expandable details
4. Expand a waypoint → see description, distance, elevation, OS Grid Ref, etc.
5. Click "Show on map" on a waypoint → map flies to that location
6. Toggle "Waypoints" in map settings → markers appear/disappear at waypoint positions
7. `npm run build` — no errors
