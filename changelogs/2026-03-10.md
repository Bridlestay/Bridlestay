# Changelog — 2026-03-10

## Session 1: Fix dragging spine points with snap OFF

### Summary
Fixed a bug where dragging a spine point with snap turned OFF left the point floating disconnected from the route line. The route line continued to follow the old snapped road geometry through the old position instead of connecting to the new position.

### Files Modified
- `components/routes/routes-map-mapbox.tsx` — Added segment clearing to the snap-OFF drag path so the route falls back to straight lines through the new position

### What Changed and Why
**Root cause**: When snap is OFF and a spine point is dragged, the `else` branch of the dragend handler only called `onWaypointUpdate` without clearing the adjacent snapped segments. The snapped segments (from when the point was originally placed with snap ON) still contained road-following geometry that passed through the OLD position. The route line builder prioritises snapped segments over waypoint positions, so the line didn't follow the moved marker.

**Fix**: The snap-OFF drag path now clears the two adjacent snapped segments (`wpIdx - 1` and `wpIdx`) before calling `onWaypointUpdate`, matching what the snap-ON path already does. The route falls back to straight lines for those segments, which is correct behaviour when snap is off.

### Route Creation Audit Findings
A full audit of route creation was performed. Additional findings:

1. **Undo after drag (medium severity)**: When undoing a drag operation, the snapped segments in `snappedSegmentsRef` don't revert to their pre-drag state because segments are stored in a ref (not React state) and aren't part of the undo history. The route may briefly show stale geometry. Not yet fixed — requires either storing segment snapshots in history or clearing all segments on undo.

2. **Circular route closing segment (noted)**: When `routeType === "circular"`, the closing segment (last waypoint back to start) is always rendered as a straight line, never snapped to roads. Additionally, `getRouteGeometry()` doesn't include the closing segment in exported geometry. Not yet fixed — requires fetching a closing segment and including it in exports.

3. **Plot, erase, insert tools**: All working correctly.
4. **Draggable flags**: Correct — markers are only draggable in plot/insert mode, not erase.
5. **getRouteGeometry**: Correctly exports snapped road-following geometry.
6. **Snap toggle mid-creation**: Correctly fetches segments from last waypoint to new click.

### Verification Steps
1. `npm run build` passes
2. Place points with snap ON → turn snap OFF → drag a middle point → route line follows the point (straight lines to neighbours)
3. Drag the start point with snap OFF → route redraws correctly
4. Drag the finish point with snap OFF → route shortens/extends correctly

## Session 2: Fix undo after drag + circular route closing segment

### Summary
Fixed two remaining route creation issues found in the Session 1 audit: (1) Undo after dragging a spine point left stale snapped road geometry on the map, and (2) circular routes drew a straight line for the closing segment instead of following roads.

### Files Modified
- `app/routes/page.tsx` — `handleUndo` now calls `clearSnappedSegments()` before restoring waypoints
- `components/routes/routes-map-mapbox.tsx` — Added `resnapTrigger` state, updated `clearSnappedSegments` to trigger re-snapping, added useEffect to fetch circular closing segment from Mapbox Directions API

### What Changed and Why

**Undo after drag (Issue 1)**:
- **Root cause**: Snapped road-following geometry is stored in `snappedSegmentsRef` (a React ref), not in React state. The undo history only stores waypoint positions. After undo, waypoint positions reverted to their old locations, but the snapped segments still contained geometry routed through the NEW (post-drag) positions. The route line prioritises snapped segments, so it drew through the wrong positions.
- **Fix**: `handleUndo` now calls `mapRef.current?.clearSnappedSegments()` which clears all segments AND bumps a `resnapTrigger` counter. The snap-fetch useEffect watches this counter, so when segments are cleared and snap is enabled, it re-fetches all segments for the restored waypoint positions. The route line temporarily shows straight lines during the fetch, then snaps back to roads.

**Circular route closing segment (Issue 2)**:
- **Root cause**: When a route is circular, the closing segment (last waypoint → first waypoint) was always drawn as a straight line. No code existed to fetch road-following geometry for this segment from the Mapbox Directions API.
- **Fix**: Added a dedicated useEffect that watches `routeType` and `waypoints`. When `routeType === "circular"` and there are 3+ waypoints, it fetches the closing segment from the Directions API (walking profile) and stores it at key `waypoints.length - 1` in `snappedSegmentsRef`. The existing rendering code (already updated in Session 1) picks up this segment and draws a road-following line back to start. The segment is automatically cleaned up when the route reverts to linear.

### Verification Steps
1. `npm run build` passes
2. **Undo after drag**: Place 3+ points with snap ON → drag a middle point → press Undo → route line re-snaps to roads through the original position
3. **Circular closing**: Create a circular route (3+ points, close back to start) → the closing segment follows roads instead of a straight line
4. **Undo circular**: Press undo to drop below 3 waypoints → route reverts to linear, closing segment removed
5. **Re-snap trigger**: Drag a point with snap ON → undo → segments re-fetch correctly for restored positions

## Session 3: Fix undo from circular + closing segment snap reliability

### Summary
Fixed two bugs: (1) Pressing undo after making a route circular didn't revert to linear — the route stayed circular and kept attaching to the last spine point. (2) The circular closing segment fetch was unreliable due to a stale cache guard.

### Files Modified
- `app/routes/page.tsx` — `handleUndo` now always reverts to linear when undoing from circular, and re-enables plotting
- `components/routes/routes-map-mapbox.tsx` — Removed stale `has(closingKey)` guard from closing segment fetch, added abort controller for cleanup

### What Changed and Why

**Undo from circular (Issue 1)**:
- **Root cause**: The condition `previousState.length < 3 && routeType === "circular"` only reverted to linear if the restored state had fewer than 3 waypoints. But circular detection doesn't add a waypoint — it just changes `routeType`. So undoing removes the last plotted point (which still has 3+ waypoints), and the route stayed circular. Also, `isPlotting` was never set back to `true` (circular detection sets it to `false`).
- **Fix**: Changed to `if (routeType === "circular")` — always revert to linear on undo from circular. Also sets `isPlotting` back to `true` so the user can continue plotting.

**Closing segment snap (Issue 2)**:
- **Root cause**: The `has(closingKey)` guard prevented re-fetching the closing segment after any waypoint change (e.g. drag). Once the closing segment was fetched, subsequent waypoint moves couldn't update it. Additionally, this guard could theoretically block the initial fetch if a stale segment existed at that key.
- **Fix**: Removed the `has(closingKey)` guard entirely. The effect now always fetches when the route is circular. Added a `cancelled` flag via the effect's cleanup function to prevent stale async responses from overwriting segments after the route changes.

### Verification Steps
1. `npm run build` passes
2. **Undo from circular**: Make a circular route → press Undo → route reverts to linear, plotting resumes
3. **Circular closing snap**: Close a route with snap ON → closing segment follows roads
4. **Drag after circular**: Drag a waypoint while circular → closing segment re-fetches for new positions

## Session 4: Snap first point to road + hide F marker on circular routes

### Summary
Fixed two UX issues: (1) The first waypoint placed with snap ON didn't snap to a road — it was placed wherever you clicked. (2) Circular routes showed the F marker floating at the last plotted spine point instead of hiding it (since start = finish on a loop).

### Files Modified
- `components/routes/routes-map-mapbox.tsx` — Added first-point snapping via Directions API, hid F marker for circular routes

### What Changed and Why

**First point snap**:
- **Root cause**: The snap condition was `if (snapEnabledRef.current && wps.length > 0)` — the `wps.length > 0` meant the very first click (empty waypoints) always skipped snapping and placed the point directly at the click position.
- **Fix**: Restructured the click handler into two branches: (1) First waypoint (`wps.length === 0`): uses a tiny-offset Directions API call to find the nearest road point and snaps the start marker there. (2) Subsequent waypoints: existing logic unchanged. Both branches have the same 15m off-road threshold — if the click is too far from a road, it places unsnapped.

**Circular F marker**:
- **Root cause**: The F marker was shown at the last waypoint regardless of route type. For circular routes this puts F floating in the middle of the loop, disconnected from the closing segment that connects back to S.
- **Fix**: Added `routeType !== "circular"` to the `isFinish` condition. On circular routes, the last spine point renders as a normal intermediate dot instead of F. The route visually loops back to S, which serves as both start and finish.

### Verification Steps
1. `npm run build` passes
2. **First point snap**: With snap ON, click near a road → S marker snaps to the road, not at the exact click position
3. **First point off-road**: With snap ON, click far from any road (>15m) → S marker places at click position (unsnapped fallback)
4. **First point no snap**: With snap OFF, click anywhere → S marker places at exact click position
5. **Circular F hidden**: Make a circular route → no F marker shown, last point is a small blue dot, line loops back to S

## Session 7: Fix comment deletion + deep-link to specific comment

### Summary
Fixed two issues: (1) Deleting comments returned 500 Internal Server Error due to RLS blocking the soft-delete update. (2) Clicking a comment notification now opens the route AND scrolls directly to the specific comment in the discussion panel with a brief green highlight.

### Files Modified
- `app/api/routes/[id]/comments/[commentId]/route.ts` — Switched to service client for the soft-delete update and comment lookup (bypasses RLS)
- `app/api/routes/[id]/comments/route.ts` — Added `&comment=${comment.id}` to notification links
- `app/routes/page.tsx` — Added `initialCommentId` state, read `comment` query param from URL, pass to drawer
- `components/routes/route-detail-drawer.tsx` — Added `initialCommentId` / `onCommentFocused` props, auto-opens discussion panel and scrolls to comment
- `components/routes/route-discussion-panel.tsx` — Added `id="comment-{id}"` attribute and transition class to each comment div

### What Changed and Why

**Comment deletion 500**: The DELETE endpoint used the regular Supabase client (which respects RLS) for the `update({ deleted_at: ... })` soft-delete. The RLS policy on `route_comments` likely doesn't allow UPDATE. Switched to `createServiceClient()` which bypasses RLS. Auth checks (ownership + admin) are still enforced in the API code before the delete.

**Comment deep-link**: Notification links now include both route and comment IDs (`/routes?route=X&comment=Y`). The routes page reads both params, opens the route detail drawer, and passes `initialCommentId` down. The drawer's useEffect detects this prop, switches to the discussion panel, then scrolls to and briefly highlights the specific comment with a green background that fades after 3 seconds.

### Verification Steps
1. `npm run build` passes
2. **Delete comment**: Open discussion → click "..." → Delete → comment is removed, toast says "Comment deleted"
3. **Comment notification deep-link**: Click a comment notification → route opens → discussion panel auto-opens → specific comment scrolls into view with green highlight

## Session 8: Deleted comment handling, notification preview, auto-cleanup

### Summary
Three improvements: (1) Clicking a notification for a deleted comment now opens the discussion panel with an info toast instead of 404. Old notification links (`/routes/{id}`) are automatically rewritten to the new format. (2) Comment notifications now show a preview of the comment text instead of just the route title. (3) Read notifications are automatically deleted after 30 days via a daily cron job, with a note on the notifications page.

### Files Modified
- `components/notifications-feed.tsx` — Rewrites old `/routes/{uuid}` links to `/routes?route={uuid}` in click handler, added Info icon import and 30-day cleanup note at bottom of feed
- `app/api/routes/[id]/comments/route.ts` — Route owner notification body changed from `route.title` to the actual comment text (truncated at 100 chars)
- `components/routes/route-detail-drawer.tsx` — Comment deep-link effect now shows toast "This comment has been deleted or removed" when the linked comment isn't found
- `app/api/cron/cleanup-notifications/route.ts` — New cron endpoint that deletes read notifications older than 30 days
- `vercel.json` — Added cleanup-notifications cron (daily at 3am UTC)

### What Changed and Why

**Old notification links**: Notifications created before the link fix still have `/routes/{id}` URLs. The click handler now detects this pattern via regex and rewrites to `/routes?route={id}` so they navigate correctly instead of hitting 404.

**Deleted comment handling**: When a notification links to a comment that no longer exists (soft-deleted), the discussion panel opens but the comment isn't in the list. The effect now checks `comments.some(c => c.id === initialCommentId)` and shows an info toast if not found.

**Comment preview**: The route owner notification was using `body: route.title` (just the route name), while reply notifications already showed the comment text. Changed to use the comment body (truncated at 100 chars) for consistency — notifications now show what was actually said.

**Auto-cleanup cron**: New endpoint at `/api/cron/cleanup-notifications` runs daily at 3am UTC. It deletes all notifications where `read = true` and `created_at < 30 days ago`. Uses the service client to bypass RLS. The notifications page now shows a small info line: "Read notifications are automatically removed after 30 days."

### Verification Steps
1. `npm run build` passes
2. **Old links**: Click an old-format notification → still navigates to correct route (no 404)
3. **Deleted comment**: Click notification for a deleted comment → discussion opens with toast "This comment has been deleted or removed"
4. **Comment preview**: Post a comment → owner notification shows the comment text, not just route title
5. **Cleanup note**: Notifications page shows "Read notifications are automatically removed after 30 days" below the feed

## Session 9: Deleted comment indicators in notifications

### Summary
Fixed the "comment deleted" toast not appearing when deep-linking to a deleted comment, and added visual indicators on the notifications page for deleted comments — "Comment removed" label in red and strikethrough on the comment preview text.

### Files Modified
- `components/routes/route-detail-drawer.tsx` — Added `commentsLoaded` state flag; changed deep-link effect condition from `comments.length > 0` to `commentsLoaded` so it fires even when all comments are deleted
- `app/api/notifications/route.ts` — Added batch check for deleted comments: parses comment IDs from `route_comment` notification links, queries `route_comments` table, flags deleted ones with `comment_deleted: true`
- `components/notifications-feed.tsx` — Added `comment_deleted` to Notification interface; shows "Comment removed" in red next to title, and strikethrough on the comment body text for deleted comment notifications

### What Changed and Why

**Toast not firing**: The deep-link effect required `comments.length > 0` to run. If the deleted comment was the only comment (or all were deleted), the array is empty and the effect body never executed. Changed to use a `commentsLoaded` boolean that's set to `true` once the comments fetch completes (even with 0 results).

**Notifications page indicator**: The GET `/api/notifications` handler now parses comment IDs from `route_comment` notification links and batch-checks against `route_comments` (where `deleted_at IS NULL`). Any comment that no longer exists gets `comment_deleted: true` in the response. The feed component shows a subtle "Comment removed" label in red-400 next to the title, and the comment preview text gets a strikethrough with reduced opacity.

### Verification Steps
1. `npm run build` passes
2. **Toast on deleted comment**: Delete a comment → click its notification → discussion opens → toast shows "This comment has been deleted or removed"
3. **Notifications page**: Notifications for deleted comments show "Comment removed" in red and strikethrough body text
