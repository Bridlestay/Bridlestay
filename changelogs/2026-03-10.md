# Changelog — 2026-03-10

## Session 1: Fix dragging spine points with snap OFF

### Summary
Fixed a bug where dragging a spine point with snap turned OFF left the point floating disconnected from the route line. The route line continued to follow the old snapped road geometry through the old position instead of connecting to the new position.

### Files Modified
- `components/routes/routes-map-mapbox.tsx` — Added segment clearing to the snap-OFF drag path so the route falls back to straight lines through the new position

### What Changed and Why
**Root cause**: When snap is OFF and a spine point is dragged, the `else` branch of the dragend handler only called `onWaypointUpdate` without clearing the adjacent snapped segments. The snapped segments (from when the point was originally placed with snap ON) still contained road-following geometry that passed through the OLD position. The route line builder prioritises snapped segments over waypoint positions, so the line didn't follow the moved marker.

**Fix**: The snap-OFF drag path now clears the two adjacent snapped segments (`wpIdx - 1` and `wpIdx`) before calling `onWaypointUpdate`, matching what the snap-ON path already does. The route falls back to straight lines for those segments, which is correct behaviour when snap is off.

### Route Creation Audit Findings
A full audit of route creation was performed. Additional findings:

1. **Undo after drag (medium severity)**: When undoing a drag operation, the snapped segments in `snappedSegmentsRef` don't revert to their pre-drag state because segments are stored in a ref (not React state) and aren't part of the undo history. The route may briefly show stale geometry. Not yet fixed — requires either storing segment snapshots in history or clearing all segments on undo.

2. **Circular route closing segment (noted)**: When `routeType === "circular"`, the closing segment (last waypoint back to start) is always rendered as a straight line, never snapped to roads. Additionally, `getRouteGeometry()` doesn't include the closing segment in exported geometry. Not yet fixed — requires fetching a closing segment and including it in exports.

3. **Plot, erase, insert tools**: All working correctly.
4. **Draggable flags**: Correct — markers are only draggable in plot/insert mode, not erase.
5. **getRouteGeometry**: Correctly exports snapped road-following geometry.
6. **Snap toggle mid-creation**: Correctly fetches segments from last waypoint to new click.

### Verification Steps
1. `npm run build` passes
2. Place points with snap ON → turn snap OFF → drag a middle point → route line follows the point (straight lines to neighbours)
3. Drag the start point with snap OFF → route redraws correctly
4. Drag the finish point with snap OFF → route shortens/extends correctly

## Session 2: Fix undo after drag + circular route closing segment

### Summary
Fixed two remaining route creation issues found in the Session 1 audit: (1) Undo after dragging a spine point left stale snapped road geometry on the map, and (2) circular routes drew a straight line for the closing segment instead of following roads.

### Files Modified
- `app/routes/page.tsx` — `handleUndo` now calls `clearSnappedSegments()` before restoring waypoints
- `components/routes/routes-map-mapbox.tsx` — Added `resnapTrigger` state, updated `clearSnappedSegments` to trigger re-snapping, added useEffect to fetch circular closing segment from Mapbox Directions API

### What Changed and Why

**Undo after drag (Issue 1)**:
- **Root cause**: Snapped road-following geometry is stored in `snappedSegmentsRef` (a React ref), not in React state. The undo history only stores waypoint positions. After undo, waypoint positions reverted to their old locations, but the snapped segments still contained geometry routed through the NEW (post-drag) positions. The route line prioritises snapped segments, so it drew through the wrong positions.
- **Fix**: `handleUndo` now calls `mapRef.current?.clearSnappedSegments()` which clears all segments AND bumps a `resnapTrigger` counter. The snap-fetch useEffect watches this counter, so when segments are cleared and snap is enabled, it re-fetches all segments for the restored waypoint positions. The route line temporarily shows straight lines during the fetch, then snaps back to roads.

**Circular route closing segment (Issue 2)**:
- **Root cause**: When a route is circular, the closing segment (last waypoint → first waypoint) was always drawn as a straight line. No code existed to fetch road-following geometry for this segment from the Mapbox Directions API.
- **Fix**: Added a dedicated useEffect that watches `routeType` and `waypoints`. When `routeType === "circular"` and there are 3+ waypoints, it fetches the closing segment from the Directions API (walking profile) and stores it at key `waypoints.length - 1` in `snappedSegmentsRef`. The existing rendering code (already updated in Session 1) picks up this segment and draws a road-following line back to start. The segment is automatically cleaned up when the route reverts to linear.

### Verification Steps
1. `npm run build` passes
2. **Undo after drag**: Place 3+ points with snap ON → drag a middle point → press Undo → route line re-snaps to roads through the original position
3. **Circular closing**: Create a circular route (3+ points, close back to start) → the closing segment follows roads instead of a straight line
4. **Undo circular**: Press undo to drop below 3 waypoints → route reverts to linear, closing segment removed
5. **Re-snap trigger**: Drag a point with snap ON → undo → segments re-fetch correctly for restored positions

## Session 3: Fix undo from circular + closing segment snap reliability

### Summary
Fixed two bugs: (1) Pressing undo after making a route circular didn't revert to linear — the route stayed circular and kept attaching to the last spine point. (2) The circular closing segment fetch was unreliable due to a stale cache guard.

### Files Modified
- `app/routes/page.tsx` — `handleUndo` now always reverts to linear when undoing from circular, and re-enables plotting
- `components/routes/routes-map-mapbox.tsx` — Removed stale `has(closingKey)` guard from closing segment fetch, added abort controller for cleanup

### What Changed and Why

**Undo from circular (Issue 1)**:
- **Root cause**: The condition `previousState.length < 3 && routeType === "circular"` only reverted to linear if the restored state had fewer than 3 waypoints. But circular detection doesn't add a waypoint — it just changes `routeType`. So undoing removes the last plotted point (which still has 3+ waypoints), and the route stayed circular. Also, `isPlotting` was never set back to `true` (circular detection sets it to `false`).
- **Fix**: Changed to `if (routeType === "circular")` — always revert to linear on undo from circular. Also sets `isPlotting` back to `true` so the user can continue plotting.

**Closing segment snap (Issue 2)**:
- **Root cause**: The `has(closingKey)` guard prevented re-fetching the closing segment after any waypoint change (e.g. drag). Once the closing segment was fetched, subsequent waypoint moves couldn't update it. Additionally, this guard could theoretically block the initial fetch if a stale segment existed at that key.
- **Fix**: Removed the `has(closingKey)` guard entirely. The effect now always fetches when the route is circular. Added a `cancelled` flag via the effect's cleanup function to prevent stale async responses from overwriting segments after the route changes.

### Verification Steps
1. `npm run build` passes
2. **Undo from circular**: Make a circular route → press Undo → route reverts to linear, plotting resumes
3. **Circular closing snap**: Close a route with snap ON → closing segment follows roads
4. **Drag after circular**: Drag a waypoint while circular → closing segment re-fetches for new positions
