## Session 10: Fix erasing start point breaks route creation

### Summary
Fixed a bug where erasing the start point (or any non-terminal waypoint) during route creation broke the route line. The root cause was that snapped segment indices were not re-indexed when a waypoint was removed, causing the route to render with wrong geometry.

### Files Modified
- `components/routes/routes-map-mapbox.tsx` â€” Added `reindexSegmentsOnRemove()` helper function; erase click handler now re-indexes segments before removing the waypoint

### What Changed and Why
1. **Snapped segment re-indexing**: Snapped segments are stored in a `Map<number, [number, number][]>` keyed by segment index (0, 1, 2...). When a waypoint is removed, the segments that reference it become invalid and all later segments need to shift down by 1. Previously, only segments beyond the new max index were pruned â€” orphaned segments in the middle caused the route line to render with wrong road-following geometry from deleted waypoints.

2. **`reindexSegmentsOnRemove()` helper**: Takes the removed waypoint's index and the segments map. Deletes the two segments that touched the removed waypoint (before and after it), then shifts all later segment keys down by 1. The gap left by the deleted segments falls back to a straight line, which is correct since the road-following path through that point no longer exists.

### Cluster Count Investigation
Investigated why cluster numbers seemed too high. The cluster system is working correctly â€” counts include ALL public routes from ALL users in the area, not just the current user's routes. The search API, my-routes merge logic, and GeoJSON feature builder all properly deduplicate and filter.
