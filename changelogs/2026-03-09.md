# Changelog — 2026-03-09

## Session 1: Waypoint "View all" and "Show on map" navigation flow

### Summary
Fixed waypoint navigation UX: "View all" and "Show on map" now close the route detail drawer, show the quick card at the bottom for easy navigation back, and enable waypoint markers on the map.

### Files Modified
- `app/routes/page.tsx` — Updated `handleEnterViewMode` and `onFlyToLocation` handlers

### What Changed and Why
1. **"View all" waypoints**: Previously closed the drawer but left no way to navigate back to the route detail. Now sets `previewRoute` so the small quick card appears at the bottom — clicking it reopens the full route detail.

2. **"Show on map" per waypoint**: Previously flew to the waypoint location but kept the drawer open, blocking the map view. Now closes the drawer, shows the quick card at the bottom, and enables the waypoints layer in map settings so markers are visible.

### Verification Steps
1. `npm run build` passes
2. Open a route detail → Waypoints section → click "View all" → drawer closes, waypoints visible on map, quick card at bottom
3. Click the quick card → route detail reopens
4. Open a route detail → click "Show on map" on any waypoint → drawer closes, map flies to waypoint, waypoint markers visible, quick card at bottom

## Session 2: Fix difficulty badge hover and elevation data display

### Summary
Fixed difficulty badges turning green on hover across all route components. Fixed elevation data fallback using wrong DB column names. Added error logging to elevation fetch.

### Files Modified
- `components/routes/route-quick-card.tsx` — Changed difficulty badge to `variant="outline"`, added border colors
- `components/routes/route-detail-drawer.tsx` — Changed difficulty badge to `variant="outline"`, fixed `elevation_gain` → `elevation_gain_m` and `elevation_loss` → `elevation_loss_m` fallback field names, added elevation fetch error logging
- `components/routes/route-bottom-sheet.tsx` — Changed difficulty badge to `variant="outline"`, added border colors
- `components/routes/saved-routes-panel.tsx` — Changed difficulty badge to `variant="outline"`, added border colors
- `components/routes/find-routes-panel.tsx` — Changed difficulty badge to `variant="outline"`, added border colors

### What Changed and Why
1. **Difficulty badge hover fix**: The Badge component's `default` and `secondary` variants include `hover:bg-primary/80` and `hover:bg-secondary/80` respectively, which turns badges green on hover. Changed all non-interactive difficulty badges to `variant="outline"` which has no hover effect. Added matching border colors to all difficulty color maps for consistent styling.

2. **Elevation field name mismatch**: The route detail drawer used `route.elevation_gain` and `route.elevation_loss` as fallback values, but the actual DB columns are `elevation_gain_m` and `elevation_loss_m`. This meant cached elevation data was never displayed as a fallback when the live API data wasn't available.

3. **Elevation error logging**: The elevation fetch was silently swallowing all errors. Added `console.error` logging for both failed HTTP responses and network errors to help debug elevation issues.

### Risk Notes
- The `variant="outline"` change is purely visual — adds a subtle border to difficulty badges that previously had none. The border colors match each difficulty level's color scheme.
- Elevation errors now appear in browser console, which helps debugging but doesn't show anything to users.

### Verification Steps
1. `npm run build` passes
2. Hover over any difficulty badge (quick card, detail drawer, bottom sheet, saved/find panels) — should NOT turn green
3. Open a route detail — check that Ascent/Descent values show numbers instead of "---" if elevation data has been cached
4. Open browser console when viewing a route — if elevation fetch fails, error will be logged

## Session 3: Visibility order, difficulty name alignment, Edit label

### Summary
Reordered visibility options (Public first, Private last), removed API difficulty mapping so user-selected names persist exactly, and renamed "Copy & Edit" to "Edit".

### Files Modified
- `components/routes/save-route-modal.tsx` — Reordered visibility radio options
- `app/api/routes/route.ts` — Removed difficulty remapping in POST handler
- `app/api/routes/[id]/route.ts` — Removed difficulty remapping in PATCH handler
- `components/routes/route-detail-drawer.tsx` — "Copy & Edit" → "Edit"
- `components/routes/route-bottom-sheet.tsx` — Added `medium`/`hard` entries + aligned colors with amber/orange scheme
- `components/routes/saved-routes-panel.tsx` — Added `medium`/`hard` entries + aligned colors
- `components/routes/find-routes-panel.tsx` — Added `medium`/`hard` entries + aligned colors
- `components/routes/nearby-routes-widget.tsx` — Added all difficulty entries for complete coverage

### What Changed and Why
1. **Visibility order**: Public is now the first option in the save modal, Private is last. Better default for route sharing.

2. **Difficulty name alignment**: The API was remapping "moderate" → "medium", "difficult"/"severe" → "hard", "unrated" → null. This caused the route card to show different labels than what the user picked during creation. Removed all remapping — the exact value the user selects now persists in the DB. All display components already support both old and new names for backward compatibility.

3. **"Copy & Edit" → "Edit"**: Simplified the menu label in the route detail options dropdown.

### Verification Steps
1. `npm run build` passes
2. Create route → visibility options show Public, Link, Private (in that order)
3. Create route with "Difficult" difficulty → route card shows "Difficult" (not "Hard")
4. Old routes with "medium"/"hard" still display correctly
5. Route detail options → shows "Edit" not "Copy & Edit"
