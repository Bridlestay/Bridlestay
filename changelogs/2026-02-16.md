# Changelog — 2026-02-16

## Elevation Graph Simplification & Waypoints/Hazards Map View Mode

### Session Summary

Simplified the elevation profile graph (line-only, no gradient fill, key-point markers) and replaced the in-drawer waypoints/hazards panels with a map-first "view mode" that hides the card and shows interactive markers directly on the map.

### Files Modified

- `components/routes/elevation-profile.tsx`
  - **Line-only graph**: Removed gradient fill (`<linearGradient>` and area path). Now renders a clean stroke-only line (`fill="none"`, `strokeWidth="1.2"`)
  - **Key-point markers**: Added useMemo computing start, end, highest, and lowest elevation points. Rendered as green-outlined circles with elevation labels (e.g., "142m") positioned above peaks and below valleys
  - **Removed header**: Removed the "Elevation" header with TrendingUp/TrendingDown icons (redundant — stats grid already shows ascent/descent)
  - Removed unused `TrendingUp`, `TrendingDown`, `Mountain` lucide-react imports

- `components/routes/route-detail-drawer.tsx`
  - **Spacing**: Added `className="mt-3"` to ElevationProfile for visual gap between stats grid and graph
  - **New props**: Added `onEnterViewMode` and `onHazardsLoaded` callback props to interface
  - **Waypoints card click**: Now calls `onEnterViewMode("waypoints")` instead of opening in-drawer panel
  - **Hazards card click**: If active hazards exist, calls `onEnterViewMode("hazards")`; if none, opens report dialog (existing behavior)
  - **Hazards callback**: `fetchHazards` now calls `onHazardsLoaded` to pass hazard data up to page level

- `components/routes/routes-map-mapbox.tsx`
  - **Waypoint hover popups**: Added `mouseenter`/`mouseleave` listeners to waypoint markers showing name + description popup
  - **Hazard marker rendering**: New `routeHazards` and `showHazards` props. Renders colored circular markers based on severity (blue=low, amber=medium, orange=high, red=critical) with warning icon. Hover shows title + type + description popup
  - **Hazard markers ref**: Added `hazardMarkersRef` for proper cleanup

- `app/routes/page.tsx`
  - **Map view mode state**: Added `mapViewMode` (`"waypoints" | "hazards" | null`) and `selectedRouteHazards` state
  - **`handleEnterViewMode`**: Hides drawer, preserves route on map, zooms to fit route bounds
  - **Floating back button**: "Back to route details" pill button shown when view mode is active, positioned top-left. Restores drawer on click
  - **Marker visibility**: Waypoints show when in view mode OR when toggle is on. Hazards show only in hazards view mode
  - **Waypoint click in view mode**: Suppressed (hover popup provides info instead of re-opening drawer)
  - Wired `onEnterViewMode` and `onHazardsLoaded` to drawer

### What Changed and Why

1. **Elevation graph was too heavy** — Gradient fill and header made the card feel dense. Simplified to a clean line with key-point markers showing start/end/highest/lowest elevations. Header removed since the stats grid already shows ascent/descent values.

2. **Waypoints/hazards panel was useless** — Opening a list inside the drawer covered the map, making spatial context impossible. Replaced with a map-first "view mode" where clicking Waypoints or Hazards hides the card, shows interactive markers on the map, and lets users hover/click markers for info. A floating back button returns to the card.

3. **No hazard visualisation on map** — Hazards only appeared as a count badge. Now renders severity-colored markers with hover popups when in hazards view mode.

### Risk Notes

- Hazards with null `lat`/`lng` are filtered out — if ALL hazards lack coordinates, view mode shows the route with no markers
- `handleEnterViewMode` uses `setTimeout(200ms)` for fitBounds to ensure drawer animation completes first
- Waypoint hover popups use `mouseenter`/`mouseleave` — won't fire on mobile touch devices (click still works)
- The fullWaypointsPanel is preserved in the drawer but no longer triggered in the normal flow (kept as fallback)

### Verification Steps

1. Open a route → elevation graph shows simple green line (no gradient fill), gap below stats grid
2. Key-point markers visible on graph: start/end elevations + highest/lowest with labels
3. Click "Waypoints" → card hides, map shows waypoint markers, hover shows name/description popup
4. Click "Hazards" (with active hazards) → card hides, map shows severity-colored hazard markers with hover info
5. Click "Hazards" with no active hazards → report dialog opens (unchanged)
6. Floating "Back to route details" button returns to card view
7. `npm run build` — no errors

## Session 2: Bug Fixes — Elevation Graph, Popup Cleanup, Show on Map

### Session Summary

Fixed three issues: elevation graph key-point labels were distorted by SVG stretching (moved to HTML overlays), waypoint popups persisted after toggling off (added explicit popup cleanup), and "Show on map" didn't dismiss the route card (now calls onDismiss).

### Files Modified

- `components/routes/elevation-profile.tsx`
  - **Key-point markers → HTML overlays**: Moved circles and elevation labels from SVG elements to absolutely positioned HTML divs. SVG `preserveAspectRatio="none"` was distorting circles into ovals and stretching text horizontally. HTML overlays use percentage positioning matching the SVG coordinate space but render with correct proportions.
  - **Hover point → HTML overlay**: Same fix for the hover tracking circle.

- `components/routes/routes-map-mapbox.tsx`
  - **Popup cleanup on marker removal**: Added `routeWaypointPopupsRef` and `hazardPopupsRef` to track all Mapbox Popup instances. Cleanup now explicitly removes all popups before removing markers — prevents orphaned popups remaining on the map when waypoints are toggled off.

- `components/routes/route-detail-drawer.tsx`
  - **"Show on map" dismisses card**: Added `onDismiss?.()` call alongside `onFlyToLocation()` when clicking "Show on map" on a waypoint. The card is dismissed (route stays on map) so the user can actually see the waypoint location.

### Verification Steps

1. Elevation graph key-point labels render as clean circles with readable text (not stretched/distorted)
2. Toggle waypoints on → hover to show popup → toggle waypoints off → popup disappears
3. Open waypoints panel → expand a waypoint → click "Show on map" → card dismisses, map flies to waypoint
4. `npm run build` — no errors

## Session 4: Hazard Completion & Route Warnings

### Session Summary

Added hazard completion (any user can mark hazards as cleared from map popups), route warnings (location-less, time-limited alerts like "route is slippery today"), and a warnings card in the drawer grid.

### Files Created

- `supabase/migrations/067_hazard_warnings_and_completion.sql` — Adds `is_warning` boolean + `resolved_by_user_id` columns to `route_hazards`, expands `hazard_type` CHECK to include warning types (slippery, muddy, weather_warning, restricted_access, other_warning), adds `active_warnings_count` to routes table, relaxes RLS UPDATE policy so any authenticated user can resolve, updates count trigger to separate hazard/warning counting

### Files Modified

- `app/api/routes/[id]/hazards/route.ts`
  - **GET**: Added `is_warning` query param filter — `?is_warning=true` returns only warnings, default returns only hazards
  - **POST**: Accepts `is_warning` flag — when true, enforces `expires_at` required, nullifies lat/lng

- `app/api/routes/[id]/hazards/[hazardId]/route.ts`
  - **PATCH**: Resolve-only requests (`{ status: "resolved" }`) now allowed for any authenticated user. Other field changes still restricted to reporter/admin. Sets `resolved_by_user_id` on resolve

- `components/routes/route-detail-drawer.tsx`
  - **WARNING_TYPES constant**: 5 warning types (Slippery Conditions, Muddy/Waterlogged, Weather Warning, Restricted Access, Other Warning)
  - **`getTimeRemaining` helper**: Formats expiry countdown (e.g., "4h 30m left", "1d 2h left")
  - **Warning state**: `warnings`, `warningDialogOpen`, `newWarning`, `submittingWarning`
  - **`fetchWarnings`**: Fetches warnings separately via `?is_warning=true`
  - **Warning alert cards**: Amber-styled cards above the grid showing active warnings with title, description, time remaining, and "Cleared" resolve button
  - **Warning title badge**: Amber "X Warning(s)" badge in title area alongside existing red hazards badge
  - **Warnings card in grid**: 3rd card in the grid (now `grid-cols-2 md:grid-cols-3`), amber hover, opens warning creation dialog
  - **Warning creation dialog**: Type picker, title, description, duration selector (2h to 1 week), no location verification, amber submit button
  - **`handleSubmitWarning`**: POSTs to hazards API with `is_warning: true` and computed `expires_at`
  - **`handleResolveHazard` updated**: Also updates warnings state, calls `onHazardResolved` callback
  - **New prop**: `onHazardResolved` for syncing resolve state with page

- `components/routes/routes-map-mapbox.tsx`
  - **New props**: `onHazardResolve`, `isAuthenticated`
  - **"Mark as Cleared" button in hazard popup**: Green button shown for authenticated users, attaches click listener after popup opens, calls `onHazardResolve` and closes popup
  - **Popup changed**: `closeButton: true` (was false) for better UX

- `app/routes/page.tsx`
  - **User auth**: Added `userId` state with Supabase auth check on mount
  - **`handleResolveHazardFromMap`**: Calls PATCH API and updates `selectedRouteHazards` state
  - **Props wired**: `onHazardResolve` + `isAuthenticated` to map, `onHazardResolved` to drawer

### What Changed and Why

1. **Hazards couldn't be marked as cleared** — `handleResolveHazard` existed but had no UI button. Added "Mark as Cleared" button to hazard map popups. Relaxed PATCH permissions so any authenticated user (not just reporter/admin) can resolve.

2. **No route-level warnings** — Added warnings as a variant of hazards (`is_warning = true` in `route_hazards` table). Warnings have no location, mandatory expiry, and display as amber alert cards in the drawer rather than map markers.

3. **Warnings needed expiry** — Duration selector (2h to 1 week) sets `expires_at`. Existing `cleanup_expired_hazards()` function already handles auto-expiry. Client shows time remaining countdown.

## Session 5: Warning UX Polish & Hazard Map Placement

### Session Summary

Polished warning UX (clickable badge, collapsible list, type-as-title, "Cleared?" action button) and added hazard reporting via map click with 10m proximity validation.

### Files Modified

- `components/routes/route-detail-drawer.tsx`
  - **Warning badge clickable**: Amber badge in title area now scrolls to `#active-warnings` section on click
  - **Collapsible warnings**: When >1 warning, only first is shown with "Show X more warnings" toggle button
  - **Type as title**: Removed Title input from warning dialog. Display uses type label (e.g., "Slippery Conditions") instead of user-entered title. `handleSubmitWarning` auto-generates title from `WARNING_TYPES` lookup
  - **"Cleared?" button**: Changed from green ghost "Cleared" (looked like a status) to amber outline "Cleared?" (looks like an action/question)
  - **Hazards card**: Updated click behavior — when no hazards, triggers `onPlaceHazard()` for map placement instead of old GPS dialog. Added "+ Report" badge button that triggers map placement regardless of existing hazard count

- `components/routes/routes-map-mapbox.tsx`
  - **New props**: `placingHazard`, `onHazardPlaced`
  - **Placement mode**: When `placingHazard` is true, sets crosshair cursor and adds map click handler
  - **Distance-to-polyline validation**: Point-to-line-segment distance calculation (flat earth approx, accurate at 10m scale). Click must be within 10m of route line
  - **Placement marker**: Red pulsing circle marker placed at valid click location
  - **Error feedback**: Toast shows distance when clicking too far from route

- `app/routes/page.tsx`
  - **New imports**: Dialog, Select, Label, Input, Textarea components + AlertTriangle, X icons
  - **HAZARD_TYPES constant**: Duplicated from drawer for page-level dialog
  - **Placement state**: `placingHazard`, `pendingHazardLocation`, `hazardCreateDialogOpen`, `newHazardData`, `submittingNewHazard`
  - **`handleStartPlacingHazard`**: Auth check, hides drawer, enters placement mode, zooms to fit route
  - **`handleHazardPlaced`**: Stores location, exits placement mode, opens creation dialog
  - **`handleCancelPlacingHazard`**: Exits placement mode, restores drawer
  - **`handleSubmitNewHazard`**: POSTs to hazards API with clicked lat/lng, updates state, returns to drawer
  - **Floating instruction bar**: "Click on the route to place a hazard" pill with cancel button, shown during placement mode
  - **Hazard creation dialog**: Type, title, details, severity — appears after valid map click
  - **Props wired**: `placingHazard` + `onHazardPlaced` to map, `onPlaceHazard` to drawer

### What Changed and Why

1. **Warning badge was not interactive** — Users expected to click the amber warning badge to jump to warnings. Added `scrollIntoView` on click with `id="active-warnings"` target.

2. **Multiple warnings cluttered the card** — Warnings stacked vertically taking too much space. Added collapsible view showing first warning with "Show X more" toggle.

3. **Warning title was redundant** — The type already describes the warning. Removed the title input; type label is used as the display title.

4. **"Cleared" looked like a completed status** — Changed to "Cleared?" with amber outline styling to look like an actionable question.

5. **No way to add hazards to routes** — Old flow required GPS proximity (500m) which doesn't work on desktop. New flow: click "+ Report" → map enters placement mode with crosshair → click on/near route (10m max) → fill in hazard details → submit. This works on all devices and provides precise placement.

### Risk Notes

- Distance-to-polyline uses flat-earth approximation — accurate at 10m scale but would be wrong at planetary distances (irrelevant here)
- `HAZARD_TYPES` is duplicated in page.tsx and drawer — could be extracted to shared constant but kept inline for simplicity
- Placement marker uses CSS animation `pulse` which may not be defined — falls back to static marker
- Map click handler during placement mode will fire on any click (including existing markers) — `e.stopPropagation()` on markers prevents conflict

### Verification Steps

1. Click amber warning badge at top of route card → scrolls to warnings section
2. With 2+ warnings → only first shows, "Show 1 more warning" button expands
3. Warning dialog has no Title field, warning cards show type as title
4. Warning "Cleared?" button is amber outline, not green status
