# Changelog — 2026-03-21

## Session 1

### Summary
Comprehensive Supabase security and performance audit. Created migration 081 to fix 11 security errors (RLS disabled on critical tables), 74+ performance warnings (`auth.uid()` → `(select auth.uid())`), and 1 Security Definer View issue.

### Changes

#### Supabase deep dive investigation
- Audited all 87 migration files to map every table, RLS status, and policy
- Discovered that migration `002_rls_policies.sql` was likely never applied to the live database — RLS was defined in the migration file but never enabled on the actual tables
- Identified 9 tables flagged by Supabase Security Advisor with RLS disabled: `users`, `host_profiles`, `properties`, `property_photos`, `availability_blocks`, `bookings`, `reviews`, `host_replies`, `route_pins`
- Found 2 additional tables (`property_facilities`, `property_equine`) that never had RLS defined in any migration
- Identified 273 occurrences of `auth.uid()` across 54 migration files causing per-row re-evaluation performance issues
- Found `moderation_queue` view flagged as Security Definer (privilege escalation risk)
- Identified 8 unused tables (never referenced in app code): `route_recordings`, `referrer_rewards`, `user_credits`, `saved_payment_methods`, `blocked_patterns`, `report_cooldowns`, `user_referral_rewards`
- Identified 3 deprecated tables still present: `reviews` (replaced by `property_reviews`/`user_reviews`), `host_replies`, `route_pins`
- Confirmed Stripe webhook and other critical API routes use `createServiceClient()` (service role, bypasses RLS) — safe to enable RLS

#### Migration 081: RLS Security & Performance Fix
- **File created**: `supabase/migrations/081_rls_security_and_performance_fix.sql`
- **What it does**:
  1. **Enables RLS on ALL 71 tables** — idempotent `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`
  2. **Drops all ~170 existing policies** and recreates them with `(select auth.uid())` instead of `auth.uid()` for the Supabase performance optimization (evaluated once per query instead of once per row)
  3. **Fixes `moderation_queue` view** — recreated with `security_invoker = true` to prevent privilege escalation
  4. **Adds RLS policies** for `property_facilities` and `property_equine` (previously had none)
  5. **Consolidates duplicate policies** on the `properties` table (had overlapping policies from migrations 002 and 004)
  6. **Adds `routes_owner_manage` policy** — route owners can now manage their own routes (previously only admins could)
- **Migration is idempotent** — safe to run multiple times

### Risk Notes
- **HIGH PRIORITY**: This migration MUST be run in the Supabase SQL editor before the next deploy. Until it's run, the tables flagged as UNRESTRICTED remain unprotected.
- The migration drops and recreates all policies. If any custom policies were added directly in the Supabase dashboard (not tracked in migrations), they will be lost.
- Enabling RLS on tables that were previously unprotected may cause queries to return empty results if the calling client doesn't have a valid auth session. All critical server-side operations use `createServiceClient()` which bypasses RLS.
- The `host_profiles` table SELECT policy restricts reads to the profile owner + admins. If any guest-facing page needs host profile data and doesn't use the service client, it may break. Verified: current usage is host-own-dashboard and admin-only contexts.
- Unused tables were NOT dropped — left for post-launch cleanup to avoid destructive changes.

#### Migration 081 idempotency fix
- **File modified**: `supabase/migrations/081_rls_security_and_performance_fix.sql`
- **What changed**:
  - Added `DROP POLICY IF EXISTS` before every `CREATE POLICY` statement (~170 policies) — migration can now be re-run safely

- **Root cause**: First run of migration partially succeeded — created some new policies but failed midway. Second run failed because those already-created policies couldn't be created again.

## Session 6

### Summary
Fixed critical bug: saving routes failed with `relation "reviews" does not exist`. Root cause: migration 082 dropped the `reviews` table, but a database trigger function (`sync_user_stats_and_badges`) still queried it on every route insert. Also fixed 8 app code references to the dropped table.

### Changes

#### Migration 083: Fix reviews reference in stats function
- **File created**: `supabase/migrations/083_fix_reviews_reference_in_stats.sql`
- **Root cause**: `sync_user_stats_and_badges()` function (migration 064) queries `FROM reviews WHERE guest_id = ...`. This function is called by `trigger_route_created` which fires `AFTER INSERT ON routes`. Since `reviews` was dropped in migration 082, every route save fails.
- **Fix**: Recreates the function to query `FROM property_reviews WHERE reviewer_id = ...` instead.

#### App code: Replace all `.from("reviews")` references
All 8 occurrences updated to use `property_reviews` with correct column names:
- `reviewer_id` instead of `guest_id`/`user_id`
- `overall_rating` instead of `rating`
- `review_text` instead of `body`
