## Session: Waypoint Suggestion System (Community Contributions)

**Time:** 11:00 - 12:00

### Summary
Implemented complete Komoot-style waypoint suggestion and approval workflow. Route owners maintain full control over official waypoints while enabling community contributions through a GPS-verified suggestion system with email notifications and admin panel approval interface.

### Architectural Decision
After `/discuss` session analyzing waypoint system architecture, clarified distinction between:
- **Drawing Waypoints:** Route path geometry points (insertWaypoint function for route creation)
- **Route Waypoints:** POIs/instructions/cautions stored in database
- **Community Input:** Suggestion system with owner approval (NOT direct creation)

Selected Komoot model: Only route authors can create official waypoints; community contributes via separate suggestion system requiring owner approval.

**File Created:** `supabase/migrations/073_waypoint_suggestions.sql`

**Table Schema:**
```sql
CREATE TABLE waypoint_suggestions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  route_id UUID NOT NULL REFERENCES routes(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  lat NUMERIC NOT NULL,
  lng NUMERIC NOT NULL,
  name TEXT NOT NULL,
  tag TEXT CHECK (tag IN ('poi', 'instruction', 'caution', 'note')),
  icon_type TEXT CHECK (icon_type IN ('viewpoint','water','hazard','parking','pub','gate','rest','historic','wildlife','bridge','ford','stile','other')),
  description TEXT,
  status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'approved', 'rejected')),
  created_at TIMESTAMPTZ DEFAULT now(),
  reviewed_at TIMESTAMPTZ,
  reviewed_by_user_id UUID REFERENCES auth.users(id) ON DELETE SET NULL,
  rejection_reason TEXT
);
```

**RLS Policies:**
- `waypoint_suggestions_public_read` - Anyone can view suggestions for public routes
- `waypoint_suggestions_auth_insert` - Authenticated users can submit (GPS verified in API layer)
- `waypoint_suggestions_owner_manage` - Route owners can update (approve/reject)
- `waypoint_suggestions_owner_delete` - Route owners can delete

**Helper Function:**
```sql
CREATE OR REPLACE FUNCTION get_pending_suggestions_count(route_uuid UUID)
RETURNS INTEGER AS $$
  SELECT COUNT(*)::INTEGER
  FROM waypoint_suggestions
  WHERE route_id = route_uuid AND status = 'pending';
$$ LANGUAGE SQL STABLE;
```

**Indexes:**
- `idx_waypoint_suggestions_route` on `route_id`
- `idx_waypoint_suggestions_user` on `user_id`
- `idx_waypoint_suggestions_status` on `status`
- `idx_waypoint_suggestions_route_status` on `(route_id, status)`

## API Endpoints Created

### 1. Waypoint Suggestions CRUD
**File Created:** `app/api/routes/[id]/waypoint-suggestions/route.ts`

**GET - List Suggestions**
- Query param: `status=pending|approved|rejected` (default: pending)
- Returns suggestions with user info (name, avatar_url)
- Ordered by `created_at DESC`

**POST - Create Suggestion**
- Validates lat/lng/name required
- Checks route exists and is public
- **Prevents owners from suggesting on own routes** (error: "Route owners should add waypoints directly")
- GPS verification handled in client (within 500m of route)
- Creates suggestion with `status='pending'`
- **Triggers async email notification** (fire-and-forget to `/waypoint-suggestions/notify`)
- Returns 201 with created suggestion + user info

### 2. Suggestion Approval/Rejection
**File Created:** `app/api/routes/[id]/waypoint-suggestions/[suggestionId]/route.ts`

**PATCH - Approve or Reject**
- Action: `approve` or `reject`
- **Owner-only permission check** (route must belong to user)
- **Approval workflow:**
  1. Fetch next `order_index` for waypoint
  2. Insert into `route_waypoints` using service client (bypasses RLS)
  3. Set `created_by_user_id` to original suggester (tracks community contributor)
  4. Update suggestion: `status='approved'`, `reviewed_at=now()`, `reviewed_by_user_id`
  5. Return both waypoint and updated suggestion
- **Rejection workflow:**
  1. Update suggestion: `status='rejected'`, optional `rejection_reason`, `reviewed_at`, `reviewed_by_user_id`
  2. Return updated suggestion
- Uses service client for atomic operations

**DELETE - Remove Suggestion**
- Suggester or route owner can delete
- Permanent removal from database

### 3. Email Notification
**File Created:** `app/api/routes/[id]/waypoint-suggestions/notify/route.ts`

**POST - Send Owner Notification**
- Fetches route owner details (email, name)
- Fetches suggestion details (tag, description)
- Sends styled email via Resend API
- Returns `emailId` on success
- Error logging with `[SUGGESTION_NOTIFY]` prefix

        setAddWaypointDialogOpen(false);
        toast.success("Waypoint added successfully!");
      }}
    />
  )}

## Workflow Diagrams

### Community Suggestion Flow
```
1. User clicks "Suggest Waypoint" near route
2. SuggestWaypointDialog opens
3. GPS check: within 500m?
   - Yes → Show form
   - No → Show distance error, "Try Again" button
4. User fills form (name, tag, icon, description)
5. Submit → POST /api/routes/[id]/waypoint-suggestions
6. API creates suggestion (status='pending')
7. API fires async email to route owner
8. User sees toast: "Waypoint suggestion submitted! The route owner will review it."
9. Owner receives email notification
```

### Owner Approval Flow
```
1. Owner opens route waypoints panel
2. Sees amber "Pending Suggestions" section with count badge
3. Reviews suggestion details (name, description, tag, suggester)
4. Clicks "Approve & Add"
5. PATCH /api/routes/[id]/waypoint-suggestions/[suggestionId] {action: 'approve'}
6. API:
   - Inserts waypoint into route_waypoints (with suggester's user_id)
   - Updates suggestion status to 'approved'
   - Returns both waypoint and suggestion
7. UI removes suggestion from pending list
8. UI adds waypoint to route waypoints
9. Toast: "Waypoint suggestion approved and added to route!"
```

### Owner Waypoint Creation Flow
```
1. Owner opens route detail drawer
2. Sees "Click map to add waypoints" hint in panel header
3. Parent component triggers map placement mode (onPlaceWaypoint callback)
4. Owner clicks map location
5. Parent passes coordinates to drawer via waypointPlacementCoords prop
6. Drawer auto-opens AddWaypointDialog with lat/lng
7. Owner fills form (name, tag, icon, description)
8. Submit → POST /api/routes/[id]/waypoints
9. API creates waypoint (succeeds via owner RLS permission)
10. Dialog closes, waypoint added to list
11. Toast: "Waypoint added successfully!"
```

## Security Considerations

**GPS Verification:**
- Client-side only (within 500m check)
- **Risk:** Users can bypass with spoofed location
- **Mitigation:** Owner approval required before waypoint appears
- **Future:** Server-side verification using route geometry

**RLS Enforcement:**
- `route_waypoints`: Owner-only via RLS ✅
- `waypoint_suggestions`: Public insert (filtered by route is_public check) ✅
- Approval workflow uses service client to bypass RLS for atomic operations ✅

**Email Spam:**
- Async fire-and-forget email
- **Risk:** User could spam suggestions to trigger emails
- **Mitigation:** Rate limiting not implemented yet
- **Future:** Add rate limit (max 5 suggestions per route per user per day)
